## 一、为什么需要 HITL

Agent 能自主调用工具、执行动作，但**自主 ≠ 可以无人监督地做任何事**。需要人介入的典型场景：

- **高风险 / 不可逆动作**：转账、下单、删库、发邮件、改生产配置——错了无法撤销。
- **低置信 / 高不确定**：模型自己也没把握，硬做不如问人。
- **合规要求**：金融、医疗、法律等场景，法规要求关键决策必须人工签字。
- **高成本动作**：超过某金额/资源阈值的操作要人批。
- **模糊意图**：用户需求不清，agent 需要澄清而非猜。

**核心判断**：HITL 不是"agent 不够强的临时补丁"，而是**把人的判断作为一等公民编排进 agent loop 的决策点**。它把"全自动"变成"人机协同"，用一点点人工换取安全、合规与可控——是高风险 Agent 落地的前提。

---

## 二、自治光谱：HITL 的位置

HITL 不是开/关，而是一条连续光谱：

| 级别 | 模式 | 人的角色 | 例子 |
|------|------|----------|------|
| L0 | Human-in-command | 人做，agent 建议 | 副驾驶式补全 |
| L1 | **Human-in-the-loop** | 关键步**等人批**才继续 | 转账前审批 |
| L2 | Human-on-the-loop | 人**旁观**可随时叫停 | 监控仪表盘 + kill switch |
| L3 | Full autonomy | 人**事后**抽查 | 低风险批处理 |

- **HITL（L1）**：人在回路**之内**，loop 会因为等人而**真的暂停**——本篇重点。
- **HOTL（L2）**：人在回路**之上**，loop 不停，人监督并保留中断权。
- 同一个 agent 不同动作可处于不同级别：读操作全自动，写操作 HITL，超大额操作双人审批。

**设计原则**：按**风险分级**分配自治级别，而不是一刀切。全自动会出事，全人工没价值。

## 三、六种 HITL 模式

| 模式 | 人做什么 | 插入点 | 典型实现 |
|------|----------|--------|----------|
| **Approval 审批** | 批准 / 拒绝某动作 | 高风险 tool_call 执行**前** | 审批门、权限弹窗 |
| **Edit 编辑** | 修改动作参数后再执行 | tool_call 执行前 | 改收件人、改金额 |
| **Input 补充输入** | 提供缺失信息 / 澄清 | agent 主动发问时 | `ask_human` 工具 |
| **Review 事后审核** | 审核最终输出再交付 | finalize **后** | 草稿审核、内容审校 |
| **Interrupt 中断接管** | 中途叫停 / 接手 | loop 任意时刻 | kill switch、人工接管 |
| **Steer 引导** | 不停 loop，给方向提示 | loop 运行中 | "重点看 X""别用 Y" |

实务里最常用前三种：**Approval（批不批）、Edit（改一改）、Input（问一问）**。后三种偏运行时监督。

### 把"问人"做成一个工具（Input 模式）

最优雅的 Input 实现：给 agent 一个 `ask_human` 工具，它需要信息时像调普通工具一样"调人"：

```json
{
  "name": "ask_human",
  "description": "当你缺少必要信息、或需要用户澄清/确认时调用。",
  "parameters": {
    "type": "object",
    "properties": {
      "question": {"type": "string"},
      "options": {"type": "array", "items": {"type": "string"}},
      "reason": {"type": "string", "description": "为什么需要问"}
    },
    "required": ["question"]
  }
}
```

这样"人"就被统一抽象成 loop 里的一个可调用资源，和别的工具一致——只是这个"工具"返回得慢（要等人）。

## 四、何时触发 HITL（升级策略）

把"什么时候该问人"写成显式策略，而不是散落在 prompt 里：

| 触发策略 | 规则 | 例子 |
|----------|------|------|
| **风险分级（risk-based）** | 动作命中高风险白名单就拦 | 写操作、删除、外发、支付 |
| **置信度（confidence-based）** | 模型/校验置信度低于阈值就问 | 检索不到、自评不确定 |
| **金额/阈值（threshold）** | 超过 $X / N 条就要批 | 转账 > 1 万、批量删 > 100 |
| **合规（policy-based）** | 法规/制度强制人审的类目 | 医疗建议、合同条款 |
| **显式检查点（checkpoint）** | 流程预设的固定审批点 | 计划批准、上线前确认 |

**反模式：审批疲劳（approval fatigue）**。问得太频繁，人会变成"无脑点同意"的橡皮图章，HITL 形同虚设。对策：只拦真正高风险的、低风险自动放行、相似请求批量审批、给清晰的风险说明让人能快速决策。

## 五、技术核心：暂停—持久化—恢复

HITL 最难的不是"弹个框等点击"，而是 loop 可能要**停很久**（人几小时甚至几天后才批），期间进程不能干等。这要求 loop 具备**可持久化的暂停与恢复**能力。

### 5.1 为什么难

- 同步阻塞等人 = 占着线程/连接干等，扛不住并发。
- 人可能离线很久，进程可能重启、机器可能换——**内存里的 loop 状态会丢**。
- 所以必须：**把 loop 状态序列化落盘 → 退出 → 收到人的决策后从断点重建并继续**。

### 5.2 标准做法

```
loop 执行到高风险动作
   ↓
① 序列化当前 run state（history、待执行动作、step 等）到持久化存储
   ↓
② 发出审批请求（Slack/邮件/工单/前端），带 run_id + 上下文 + 风险说明
   ↓
③ 退出/挂起，释放资源（不阻塞）
   ── 人在几分钟~几天后批准/拒绝/编辑 ──
   ↓
④ 凭 run_id 加载 state，注入人的决策，从断点继续 loop
```

关键工程点：
- **幂等恢复**：恢复后不能把已执行的副作用重复执行一遍（见《Agent工程细节与可靠执行》的幂等/idempotency_key）。
- **超时与默认动作**：人迟迟不回怎么办？要设超时 + 默认（拒绝/升级/降级），不能无限挂起。
- **恢复后状态一致性**：人批准时世界可能已变（库存没了、汇率变了）——恢复后要重新校验前提，而不是盲目执行旧动作。
- **审计留痕**：谁、何时、批了什么、基于什么上下文，全要记录（合规刚需）。

## 六、主流实现对比

### 6.1 LangGraph（HITL 的标杆实现）

把"暂停-持久化-恢复"做成一等能力：`interrupt()` + checkpointer。

```python
from langgraph.types import interrupt, Command

def approval_node(state):
    decision = interrupt({                 # 暂停并把内容抛给客户端，state 由 checkpointer 落盘
        "action": state["pending_action"],
        "risk": "high",
    })
    if decision == "approve":
        return {"approved": True}
    return {"approved": False}

# 编译时必须带 checkpointer（持久化）
app = graph.compile(checkpointer=PostgresSaver(...))
config = {"configurable": {"thread_id": "run-123"}}

app.invoke(inputs, config)                 # 跑到 interrupt 处暂停返回
# ……人在任意时间决策……
app.invoke(Command(resume="approve"), config)  # 凭 thread_id 从断点恢复
```

- `interrupt(value)`：暂停图、surface 给人、state 持久化。
- `Command(resume=...)`：注入人的决策、从断点继续。
- 也支持 `interrupt_before=["tool_node"]` 静态断点。
- **注意**：恢复时包含 `interrupt` 的那个节点会**从头重跑**，`interrupt` 之前的代码会再执行一次——副作用要放在 interrupt 之后或做幂等。

### 6.2 Claude Code / Claude Agent SDK

产品级权限系统，本质是 Approval 模式：

- 工具执行前弹**权限确认**；`canUseTool` 回调让你自定义批准逻辑。
- **Hooks**（`PreToolUse`）可在工具调用前拦截、放行或阻止。
- `allowedTools` / `permission mode` 配置哪些自动放行、哪些要问。

### 6.3 OpenAI Agents SDK

工具可标注**需要审批**，`Runner` 运行到该工具时产生 interruption，由你批准/拒绝后再续跑（暂停态可序列化以支持异步审批）。

### 6.4 AutoGen

用 `UserProxyAgent` 代表人，`human_input_mode` 控制介入频率：

```python
user = UserProxyAgent("human", human_input_mode="ALWAYS")  # ALWAYS / NEVER / TERMINATE
```

- `ALWAYS`：每轮都要人输入；`TERMINATE`：仅在该结束时问；`NEVER`：全自动。

### 6.5 CrewAI

任务级开关：

```python
task = Task(description="...", human_input=True)  # 任务执行后请人反馈/确认
```

### 6.6 HumanLayer / Temporal（专用基础设施）

- **HumanLayer**：专做 HITL 审批的 SDK/服务，把审批路由到 Slack/邮件，用装饰器包住高风险函数（`@hl.require_approval()`），天然支持异步长等待。
- **Temporal** 等持久化工作流引擎：把 agent 当 durable workflow，**原生支持等待数天的人工信号**，进程重启不丢状态——是"长时间挂起"最稳的底座。

### 6.7 对照速查

| 实现 | 主打模式 | 暂停-恢复机制 | 异步长等待 | 备注 |
|------|----------|----------------|------------|------|
| LangGraph | Approval/Edit/Input | `interrupt()` + checkpointer + `Command(resume)` | ✅ | HITL 标杆，可编程 |
| Claude Code/SDK | Approval | 权限弹窗 / `canUseTool` / hooks | 交互态为主 | 产品级开箱 |
| OpenAI Agents SDK | Approval | 工具 `needs_approval` + interruption | ✅（可序列化） | 官方轻量 |
| AutoGen | Input/Approval | `UserProxyAgent` + `human_input_mode` | 偏同步 | 对话式 |
| CrewAI | Review/Input | `human_input=True` | 偏同步 | 任务级 |
| HumanLayer | Approval | 装饰器 + 外部审批路由 | ✅ | 专用 HITL 基建 |
| Temporal | 任意 | durable workflow + signal | ✅✅ | 长任务最稳底座 |

**观察**：交互式 CLI/对话（Claude Code、AutoGen）多用**同步等待**；要扛"等几天 + 进程可重启"的生产审批，必须是**持久化 + 异步信号**那一派（LangGraph + 持久 checkpointer、HumanLayer、Temporal）。

## 七、设计要点与最佳实践

1. **按风险分级，别一刀切**：读自动、写审批、超大额双人审。把触发规则写成显式策略。
2. **防审批疲劳**：只拦真高风险，低风险放行，相似请求批量批，给足上下文让人秒批。
3. **给人决策所需的一切**：动作是什么、为什么、风险、影响范围、建议——审批 UX 决定 HITL 有没有用。
4. **超时要有默认动作**：拒绝 / 升级 / 降级，绝不无限挂起。
5. **幂等恢复**：恢复后已执行的副作用不能重做（idempotency_key、执行前先查状态）。
6. **恢复后重新校验前提**：人批准时世界可能已变，执行前再确认一次。
7. **全程审计留痕**：谁、何时、基于什么、批了什么——合规与复盘都需要。
8. **把"问人"工具化**：`ask_human` 让人成为 loop 里统一的可调用资源。

## 八、失败模式

| 失败模式 | 表现 | 对策 |
|----------|------|------|
| 审批疲劳 / 橡皮图章 | 人无脑点同意，HITL 失效 | 只拦高风险 + 清晰风险说明 + 批量 |
| 人成为瓶颈 | 大量请求堆在人那 | 风险分级、自动放行低风险、并行审批 |
| 恢复重复副作用 | 批准后重复转账/发邮件 | 幂等 + 执行前查状态 |
| 陈旧上下文 | 用过期前提执行旧动作 | 恢复后重新校验前提 |
| 无限挂起 | 人不回，loop 永远卡 | 超时 + 默认动作 |
| 竞态 | 多人/多端同时批同一动作 | 锁 / 单次消费 / 状态机 |
| 上下文不足 | 人看不懂在批什么，瞎批 | 审批请求带完整上下文与风险 |

## 九、Go（eino）审批门示例

手写 loop 里插一个审批门：高风险工具调用前同步等待批准（生产中把 `awaitApproval` 换成持久化 + 异步信号）：

```go
var highRisk = map[string]bool{"transfer": true, "delete": true, "send_email": true}

func loopWithApproval(ctx context.Context, chat model.ToolCallingChatModel,
	tools map[string]tool.InvokableTool, approve func(schema.ToolCall) bool,
	history []*schema.Message, maxSteps int) (string, error) {

	for step := 0; step < maxSteps; step++ {
		resp, err := chat.Generate(ctx, history)
		if err != nil {
			return "", err
		}
		history = append(history, resp)
		if len(resp.ToolCalls) == 0 {
			return resp.Content, nil
		}
		for _, call := range resp.ToolCalls {
			// 审批门：高风险动作执行前等人批
			if highRisk[call.Function.Name] && !approve(call) {
				history = append(history, schema.ToolMessage(
					"用户拒绝了该操作，请改用其他方式或终止。", call.ID))
				continue // 被拒：把结果回填让模型改道
			}
			out, e := tools[call.Function.Name].InvokableRun(ctx, call.Function.Arguments)
			if e != nil {
				out = "ERROR: " + e.Error()
			}
			history = append(history, schema.ToolMessage(out, call.ID))
		}
	}
	return "", errors.New("max steps reached")
}
```

> `approve` 在 CLI 里可以是同步读输入；在生产里应替换为「落盘 run state → 发审批请求 → 退出 → 凭 run_id 异步恢复」，参考 LangGraph `interrupt` / Temporal signal。

## 十、思考题（含解答）

### Q1：HITL 和 HOTL（human-on-the-loop）有什么区别？
HITL（in）人在回路**之内**，关键步会**真的暂停**等人批准才继续；HOTL（on）人在回路**之上**，loop 不停，人旁观并保留随时中断（kill switch）的权力。前者用于高风险/不可逆动作，后者用于需要监督但不必每步卡顿的场景。同一 agent 可按动作风险分别采用。

### Q2：为什么 HITL 的难点是"暂停-持久化-恢复"而不是"弹个框"？
因为人可能几小时甚至几天后才响应，期间不能阻塞线程干等，且进程可能重启、机器可能更换，内存里的 loop 状态会丢。所以必须把 run state 序列化落盘、释放资源退出，收到决策后凭 run_id 从断点重建继续——这要求 loop 本身可持久化、可恢复、且恢复幂等。

### Q3：恢复执行时最容易踩的两个坑是什么？
①重复副作用：恢复后把已执行的转账/发邮件又做一遍——要靠幂等（idempotency_key）或执行前查状态。②陈旧上下文：人批准时世界已变（库存没了、价格变了），盲目执行旧动作出错——恢复后要重新校验前提再执行。

### Q4：审批疲劳是什么，怎么破？
问得太频繁导致人无脑点同意，HITL 退化成橡皮图章。破法：按风险分级，只拦真正高风险/不可逆/大额的动作，低风险自动放行；相似请求批量审批；审批请求附清晰的动作、风险、影响说明让人能快速且有依据地决策。

### Q5：把"问人"做成 `ask_human` 工具有什么好处？
把"人"统一抽象成 loop 里的一个可调用资源，和其它工具一致——agent 缺信息时像调工具一样"调人"，无需为 HITL 改循环骨架。唯一区别是这个"工具"返回慢（要等人），因此要配合异步暂停-恢复与超时默认动作。

## 十三、核心要点速记

1. HITL 是把人的判断作为一等公民编排进 loop 的决策点，不是临时补丁。
2. 自治分光谱，按动作**风险分级**分配级别：读自动、写审批、大额双人审。
3. 六种模式，最常用 Approval（批）/ Edit（改）/ Input（问）。
4. 触发要写成显式策略（风险/置信度/阈值/合规/检查点）。
5. 技术核心是**暂停-持久化-恢复**，难在"人可能几天后才回 + 进程会重启"。
6. 恢复三件套：幂等、超时默认动作、重新校验前提。
7. 最大反模式是审批疲劳——只拦真高风险，给足上下文。
8. 要扛异步长等待，用持久化派（LangGraph+checkpointer / HumanLayer / Temporal），别用纯同步阻塞。
