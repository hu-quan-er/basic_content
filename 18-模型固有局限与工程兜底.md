# 模型固有局限与工程兜底：哪些地方必须靠工程补

> 目标：把"因为 LLM 的原理/训练方式带来的固有局限，必须在系统层做工程兜底"的场景做成一张全景图。上下文窗口（见《[17-上下文窗口的模型原理](./17-上下文窗口的模型原理.md)》）只是其中一格，本篇系统梳理其余各格：每个局限讲清「原理根因 → 表现 → 工程兜底 → 详见哪篇」。

---

## 一、为什么需要这张全景图

把 LLM 当成一个**高能力但有固定缺陷的概率组件**，是做 Agent 工程的基本心态。它的很多"毛病"不是 bug、不是 prompt 没调好，而是**下一 token 预测 + 训练分布 + Transformer 架构**带来的原理性后果——只能在外面套工程，不能指望模型自己改掉。

**核心判断**：模型层的固有局限大致归为七类（知识、能力、可控性、上下文、对齐、性能、可审计），而工程兜底也收敛成七种反复出现的模式。看懂"哪类毛病 → 哪种兜底"，就掌握了 Agent 系统设计的主干。

## 三、知识与事实类

### 3.1 幻觉（事实性）
- **根因**：LLM 训练目标是最大化下一 token 的似然，优化的是"读起来对不对劲"而非"是不是事实"；参数里没有可查证的真相源。
- **表现**：编造引用、虚构 API、把不确定的事说得很笃定。
- **工程兜底**：① 外接真相源——RAG 把权威文档塞进上下文做 grounding；② 要求带引用、可溯源；③ 关键结论用程序或第二模型校验；④ 降低开放生成、收紧到"只根据给定材料回答"。
- 详见《RAG全景》《评估与可观测性》。

### 3.2 知识时效性（cutoff）
- **根因**：模型参数是某个时间点训练数据的"冻结快照"，之后的世界它不知道。
- **工程兜底**：把"需要最新/私有数据"的部分外包给工具（搜索、数据库、内部 API），让模型只负责推理和组织，不负责"记住事实"。
- 详见《工具使用与MCP》。

### 3.3 置信度校准差
- **根因**：模型对"自己不知道"缺乏可靠的内部信号，RLHF 还会强化"给个答案"的倾向，导致过度自信。
- **工程兜底**：设弃权机制（低置信/检索不到就说不知道）、多次采样看一致性（self-consistency）、用外部验证器而非模型自评来判对错。

## 四、能力边界类

### 4.1 精确计算 / 数学
- **根因**：模型不是在做算术，而是在按 token 模式拼"看起来对的数字"，位数一多就错。
- **工程兜底**：把计算交给**代码解释器 / 计算器工具**（Program-of-Thoughts：让模型写代码、由运行时算），模型只负责列式子。
- 详见《工具使用与MCP》。

### 4.2 多步推理易错（误差累积）
- **根因**：自回归生成无法回溯，前一步错了后面顺着错，长链条错误累积。
- **工程兜底**：CoT 显式分步、任务分解、加验证器/critic、用 reasoning 模型（test-time compute）、self-consistency 多路投票。
- 详见《Prompt工程与推理范式》《Reasoning Models专题》。

### 4.3 Tokenization 副作用
- **根因**：模型看到的是 subword token，不是字符或数值。所以"数单词里有几个 r""反转字符串""大数加减"这类字符/数字级任务天生吃力。
- **工程兜底**：识别出这类任务直接转工具（字符串处理、计算），不要硬让模型做。

## 五、输出可控性类

### 5.1 非确定性
- **根因**：采样解码（temperature/top-p）天然带随机；同一输入多次调用结果可能不同。
- **工程兜底**：要可复现就 `temperature=0` + 固定 seed；要稳健就在输出侧加 schema 校验 + 失败重试，而不是假设模型每次都对。
- 详见《Agent工程细节与可靠执行》。

### 5.2 结构化输出不可靠
- **根因**：自由文本生成不保证语法合法，复杂 JSON 容易缺括号、多逗号、字段漂移。
- **工程兜底**（可靠性递增）：prompt 约束 < JSON mode < function calling / 工具调用 < 约束解码（grammar/GBNF 强制合法）；**任何一层都要再叠加一道 schema 校验 + 自纠重试**。
- 详见《工具使用与MCP》《Agent工程细节与可靠执行》。

### 5.3 重复 / 退化生成
- **根因**：高似然路径有时是自我重复的环，贪心解码尤其容易陷进去。
- **工程兜底**：解码层加 repetition/frequency penalty、设 stop 序列；agent loop 层做动作指纹死循环检测（见《Agent Loop专题》子文档）。

### 5.4 Prompt 敏感 / 脆弱
- **根因**：模型对措辞、示例顺序、格式高度敏感，小改动可能大幅改变输出。
- **工程兜底**：prompt 模板化与版本管理、用 few-shot 锚定格式、建回归评测集（改 prompt 必须跑评测防退化），把 prompt 当代码管理。
- 详见《评估与可观测性》。

## 六、上下文与状态类

### 6.1 上下文窗口 & 位置偏置
- 已在《[17-上下文窗口的模型原理](./17-上下文窗口的模型原理.md)》详述：自注意力 O(n²) + KV Cache 决定硬上限，Lost in the Middle 决定软退化。
- **工程兜底**：截断 / 压缩（compaction）/ RAG / 外置记忆；关键信息放首尾。

### 6.2 长程一致性 / 状态遗忘
- **根因**：模型本身无持久状态，早期设定的约束、变量会随上下文增长被"挤出"或淡化，导致前后矛盾、违反先前约定。
- **工程兜底**：把关键约束、已知事实、中间结论固化到**结构化 run state**（而非只靠 message 历史），每轮按需重注入；长任务用 checkpoint。
- 详见《Agent工程细节与可靠执行》《记忆与上下文工程》。

## 七、对齐与安全类

### 7.1 Prompt Injection / 越权
- **根因**：模型把上下文里的所有文本一视同仁地当指令——system、user、检索到的网页、工具返回之间**没有内建的信任边界**。所以一段被污染的工具结果就能劫持 agent。
- **工程兜底**：权限隔离（agent 只拿最小权限）、输入/输出护栏、把"数据"和"指令"分离、危险动作 HITL 审批、工具执行沙箱、必要时 dual-LLM 模式。
- 详见《安全与对齐》。

### 7.2 Sycophancy（迎合）
- **根因**：RLHF 用人类偏好训练，副作用是模型倾向于附和用户、迎合诱导。
- **工程兜底**：用独立验证器而非模型自评、避免诱导式/带预设答案的提问、评估用 rubric 而非开放打分（防 judge 也被迎合）。

### 7.3 偏见 / 有害内容
- **根因**：训练语料本身含社会偏见与有害内容。
- **工程兜底**：输入输出内容过滤、红队测试、对齐微调、敏感场景人审。

## 八、性能与成本类

### 8.1 延迟
- **根因**：自回归逐 token 生成，本质是串行的，输出越长越慢；agent loop 还要多轮叠加。
- **工程兜底**：流式输出（先出首 token）、投机解码（小模型猜、大模型验）、prompt caching、并行工具、小模型分流。
- 详见《工程化：性能、成本、可靠性》《Agent Loop专题》第十节。

### 8.2 成本
- **根因**：按 token 计费，大模型单价高；agent loop 历史累积让后面的轮越来越贵。
- **工程兜底**：模型路由/级联（简单的走便宜模型）、prompt caching（复用 KV cache）、上下文压缩、批处理。

---

## 九、可审计类

### 9.1 可复现 / 可调试差
- **根因**：非确定性 + 模型内部是黑盒，"为什么这轮这么干"无法从权重读出。
- **工程兜底**：逐步 trace（每轮输入/输出/工具/耗时/cost）、固定 seed、维护评测集做回归，把不可解释的模型行为变成可回放的系统行为。
- 详见《评估与可观测性》。

## 十、工程兜底的七种通用模式

把上面十九个局限的对策抽象一下，会发现**工程兜底反复收敛成这七种模式**——记住这七种，比记十九个细节更有用：

| 模式 | 解决哪类局限 | 本质 |
|------|--------------|------|
| **外接真相源**（RAG / 工具 / 代码执行） | 幻觉、时效、数学、字符级任务 | 模型不擅长的事，外包给确定性系统 |
| **约束生成**（function calling / 约束解码 / schema） | 结构化输出、格式 | 把输出空间限制到合法集合 |
| **校验 + 重试 / 自纠**（validator / self-consistency） | 非确定、推理错、假完成 | 不信单次输出，验证后再用 |
| **结构化状态外置**（run state / memory / checkpoint） | 上下文上限、长程遗忘 | 把"记忆"从模型搬到系统 |
| **信任边界与隔离**（权限 / 护栏 / 沙箱 / HITL） | 注入、越权、有害 | 给无边界的模型加边界 |
| **路由与缓存**（级联 / caching / 投机解码） | 成本、延迟 | 用便宜/复用换昂贵的模型调用 |
| **可观测**（trace / seed / eval） | 不可复现、难调试 | 把黑盒行为变成可回放数据 |

**核心结论**：Agent 工程的主线，就是用这七种工程模式去补 LLM 七类原理性缺陷。"模型不行就换更强模型"只能缓解部分能力类问题，可控性、安全、成本、可审计这些必须靠系统设计兜底——这也是 Agent 工程师的核心价值所在。

## 十一、思考题（含解答）

### Q1：为什么说幻觉、数学差、知识过时这三类问题可以用同一种思路解决？
它们本质都是"模型不该负责的事被迫让它负责"。统一思路是**外接真相源**：事实交给 RAG、实时数据交给搜索/API、精确计算交给代码解释器。模型只负责理解意图、编排工具、组织结果，不负责"记住事实"和"亲自算"。

### Q2：结构化输出为什么光靠 prompt 不够？
因为自由生成不保证语法合法，prompt 只是"请求"不是"约束"，复杂 JSON 仍会崩。可靠性递增的手段是 JSON mode → function calling → 约束解码（grammar 强制只能生成合法 token），而且**任何一层之上都要再加 schema 校验 + 自纠重试**——把"希望它对"变成"验证它对"。

### Q3：Prompt Injection 的根因和上下文窗口的根因，性质上有什么不同？
上下文窗口是**架构成本**问题（O(n²)、KV cache 是物理约束）；Prompt Injection 是**对齐/设计**问题——模型对上下文里 system/user/工具文本一视同仁，没有内建信任边界。前者靠压缩/外置记忆缓解，后者必须靠权限隔离、护栏、沙箱在系统层强加边界，二者不能互换手段。

### Q4：哪些局限"换更强的模型"能解决，哪些不能？
能力类（数学、推理深度、部分幻觉）会随模型变强而缓解；但**可控性（非确定、格式）、安全（注入、越权）、成本/延迟、可审计**这些是原理性的，再强的模型也需要约束生成、信任边界、路由缓存、trace 来兜底。所以 Agent 工程不会因为模型变强而消失。

### Q5：为什么"用模型自评来判断对错"经常不可靠？
两个原因叠加：置信度校准差（模型不知道自己不知道）+ sycophancy（倾向附和）。所以校验应尽量用**确定性手段或独立验证器**（程序校验、测试、第二模型带 rubric），能用程序验证的地方不要用 LLM judge。

## 十四、核心要点速记

1. 模型的很多"毛病"是原理性的（下一 token 预测 + 训练分布 + 架构），不是 prompt 没调好。
2. 局限归七类：知识、能力、可控性、上下文、对齐、性能、可审计。
3. 兜底收敛成七模式：外接真相源、约束生成、校验重试、状态外置、信任边界、路由缓存、可观测。
4. 幻觉/数学/时效是同一招——外接真相源，让模型别负责它不该负责的事。
5. 结构化输出要"验证它对"而不是"希望它对"。
6. 上下文上限是架构成本问题，Prompt Injection 是信任边界问题，手段不能互换。
7. 换更强模型能缓解能力类问题，但可控/安全/成本/可审计永远需要工程兜底——这是 Agent 工程师的价值所在。
