# 上下文窗口的模型原理：为什么有窗口、为什么会退化

> 目标：从 Transformer 架构层面解释"上下文窗口"这件事——为什么存在硬上限、为什么窗口没满效果也会变差、长窗口的成本来自哪里，以及业界用什么架构手段去扩展。这是《记忆与上下文工程》和《Agent Loop 专题》里"上下文管理"为什么必要的底层原因。

---

## 一、先把"上下文窗口问题"拆成两个

很多人把"上下文窗口"只理解成一个数字（4k / 128k / 1M）。在模型原理层，它其实是两个不同的问题：

| 问题 | 表现 | 根因所在 |
|------|------|----------|
| **硬上限**：为什么不能无限长？ | 超过某长度直接报错 / 崩坏 | 自注意力的 O(n²) 成本 + KV cache 显存 + 位置编码训练长度 |
| **软退化**：窗口没满、信息也都塞进去了，为什么效果还变差？ | Lost in the Middle、长文召回率下降 | 注意力稀释 + 训练分布偏置 + 位置编码远端衰减 |

**核心判断**：上下文工程（截断、压缩、外置记忆、把关键信息放首尾）不是工程口味，而是对抗 Transformer 这两个架构性约束的必然结果。理解了这一节，才知道 agent loop 里的 compaction 到底在跟什么作斗争。

---

## 二、硬上限之一：自注意力的 O(n²)

Transformer 的核心是自注意力：

```
Attention(Q, K, V) = softmax(QKᵀ / √d) · V
```

- Q、K、V 都是 `n × d`（n = token 数，d = 维度）。
- `QKᵀ` 是一个 **n × n 的注意力矩阵**——每个 token 要对所有 token 算一次相关性。
- 因此：**计算量 ∝ n²·d，注意力矩阵显存 ∝ n²**。

后果很直接：

| 序列长度 n | 相对计算/显存 |
|-----------|---------------|
| 1k | 1× |
| 2k | 4× |
| 8k | 64× |
| 32k | 1024× |

n 翻倍，成本翻四倍。这就是训练阶段长上下文为什么这么贵的根本原因——不是工程没优化，是注意力机制本身就是平方阶的。

**核心结论**：自注意力让每个 token 都"看见"所有 token，这是它强的原因，也是 n² 成本的来源；上下文长度和算力/显存是平方关系，不是线性。

## 三、硬上限之二：KV Cache（推理期真正的瓶颈）

训练之外，**推理（自回归生成）阶段的瓶颈其实是显存里的 KV Cache**。

### 3.1 为什么要缓存 KV

自回归生成一次吐一个 token，每生成一个新 token，都要让它对前面所有 token 做注意力。如果每步都重算前面所有 token 的 K、V，就是巨大的重复计算。所以把每层、每个 token 的 K、V **缓存起来**，新 token 直接复用。

代价：这块缓存的显存随序列长度线性增长。

### 3.2 KV Cache 显存公式

```
KV显存 = 2 × n_layers × n_kv_heads × head_dim × seq_len × batch × bytes_per_elem
         ↑K和V
```

举例（Llama-2-70B：80 层、head_dim=128、fp16）：

| 注意力类型 | KV 头数 | 每 token KV | 100k tokens |
|------------|---------|-------------|-------------|
| MHA（多头，64 头） | 64 | ≈ 2.5 MB | ≈ 244 GB |
| GQA（分组，8 头） | 8 | ≈ 0.31 MB | ≈ 30 GB |

光是 100k 上下文的 KV cache 就要吃几十 GB 显存——**而且常常先于算力把卡打满**。这解释了为什么长上下文推理"贵且慢"，也解释了 MQA/GQA（减少 KV 头数）为什么是标配。

**核心结论**：长上下文在推理期的硬约束往往不是算力，而是 KV cache 显存，它随 token 数线性增长 × 层数 × 头数。

## 四、硬上限之三：位置编码与训练长度的"隐形天花板"

注意力本身是**置换不变**的（打乱 token 顺序，attention 结果不变），模型靠**位置编码**才知道顺序。位置编码方式决定了"能不能超出训练长度"。

| 位置编码 | 机制 | 外推能力 |
|----------|------|----------|
| 绝对位置（学习式 / 正弦） | 给每个绝对位置一个向量 | 差：超出训练见过的最大位置就"没见过"，直接崩 |
| **RoPE**（旋转位置编码） | 用旋转角表示相对位置 | 中：能有限外推，但超太多会频率混叠、远端失真 |
| **ALiBi** | 给注意力分数加"距离越远惩罚越大"的线性偏置 | 较好：天然支持比训练更长的序列 |

**长度外推技术**（让"超长位置"压回训练见过的范围）：
- **Position Interpolation (PI)**：把位置坐标线性压缩进训练范围。
- **NTK-aware scaling / YaRN**：按频率非均匀缩放 RoPE，兼顾远近。

**关键认知**：标称 128k 的窗口 ≠ 128k 都同等好用。很多长窗口是靠外推技术"硬撑"出来的，**越靠近窗口末端、超出原始训练长度越多，质量越打折**。这也是下一节"软退化"的一部分来源。

---

## 五、软退化：为什么窗口没满也会变差（Lost in the Middle）

即使没超窗口、信息全塞进去了，模型对**放在中间**的关键信息召回率也明显低于放在**首尾**——这就是 Liu et al. (2023) 的 *Lost in the Middle*，性能呈 **U 型曲线**。

原理层的几个解释：

1. **softmax 注意力被稀释**：注意力权重经 softmax 在所有 key 上分摊，n 越大，平均每个远距离 token 能分到的权重越小，中段信息容易被"淹没"。
2. **训练分布偏置**：训练语料里超长程依赖的样本本就稀少，模型对"隔很远的两个 token 之间的关联"学得弱。
3. **位置编码远端衰减**：RoPE 等对远距离相对位置的区分度随距离下降。
4. **首尾天然突出**：因果注意力下，靠前的 token 被后续所有 token 反复 attend（类似"锚点"），靠后的 token 有近因优势——夹在中间的最吃亏。

**核心结论**：上下文窗口大 ≠ 能均匀利用。中段是注意力的"洼地"，所以工程上要主动把关键信息放首尾、压缩或外置中段——这正是 Lost in the Middle 给上下文工程的直接指令。

## 六、把窗口做长的架构方向

业界扩展上下文，本质是在两条路上想办法：**让 n² 变便宜** 或 **改掉 n² 这个复杂度阶**。

| 方向 | 代表 | 思路 | 代价/权衡 |
|------|------|------|-----------|
| 稀疏注意力 | Longformer / BigBird | 只算"局部 + 少量全局"，跳过大部分 pair | 近似全注意力，可能丢远程精度 |
| 滑动窗口注意力 | Mistral | 每层只看固定窗口，靠叠层扩大有效感受野 | 单层看不到全局 |
| IO 感知精确注意力 | **FlashAttention** | 不改 O(n²) 算力，但**不实体化 n×n 矩阵**，显存从 O(n²) 降到 O(n)，大减 HBM 搬运 | 精确、无近似；只是更快更省，不改复杂度阶 |
| KV 头压缩 | MQA / GQA | 多个 query 头共享少量 KV 头，砍 KV cache | 轻微质量损失 |
| 注意力沉降 | StreamingLLM | 保留开头几个"sink token"稳定分布，支撑近乎无限的流式 | 不是真的全局记忆 |
| 线性注意力 / SSM | Mamba / RWKV / RetNet | 用"状态"替代全注意力，推理 O(n)、显存常数 | 长程精确检索能力与 Transformer 有差异 |
| 序列并行 | Ring Attention | 把超长序列切分到多卡协同算注意力 | 需要分布式 |

**关键认知**：没有"免费的无限上下文"。每种扩展都在 **成本 ↔ 精确度/全局性** 之间做权衡。FlashAttention 是少数"纯赚"的（精确且更省），但它降的是显存搬运，不改 O(n²) 算力本身。

## 七、回到 Agent Loop：原理如何决定工程

把这一节的原理映射回《Agent Loop 专题》第五节的上下文管理，就全通了：

| 模型原理 | 对应的 loop 工程对策 |
|----------|---------------------|
| 硬上限（n²、KV 显存、训练长度） | 必须截断 / 压缩（compaction）/ 外置记忆，不能无脑拼接历史 |
| 软退化（Lost in the Middle） | 即使没满也要把关键信息放首尾、压缩中段、固化到结构化 state |
| KV cache 随 n 线性、算力随 n² | 解释了 loop 成本为何随轮数近似平方增长 |
| KV cache 可复用 | **prompt caching 的本质就是复用稳定前缀已算好的 KV**——这是 loop 场景省钱的关键 |

**核心结论**：agent loop 里"每轮只给决策这一步所需的最小充分信息"不是洁癖，是因为模型的注意力既算不起也用不好过长的上下文。上下文工程 = 在架构硬约束下做信息调度。

## 八、思考题（含解答）

### Q1：为什么自注意力是 O(n²)？
因为每个 token 都要对序列里所有 token 算注意力分数，`QKᵀ` 形成一个 n×n 矩阵，计算量 ∝ n²·d、注意力矩阵显存 ∝ n²。n 翻倍成本翻四倍。这是"每个 token 看见所有 token"这一机制的直接代价。

### Q2：推理时长上下文为什么贵且慢？
瓶颈是 KV Cache 显存：为避免重复计算，每层每个 token 的 K、V 都被缓存，显存 = 2 × 层数 × KV头数 × head_dim × seq_len × batch × bytes，随序列长度线性增长，常常先于算力打满显存。所以长上下文推理贵在显存，MQA/GQA、KV 量化、PagedAttention 都是为压它。

### Q3：标称 128k 窗口是不是 128k 都好用？
不是。两个原因：① 很多长窗口靠 RoPE 外推（PI/NTK/YaRN）硬撑，越超出原始训练长度、越靠窗口末端质量越打折；② Lost in the Middle——放中段的信息召回率明显低于首尾。所以"窗口能装"和"装进去能被用好"是两回事。

### Q4：Lost in the Middle 的原理层解释是什么？
softmax 注意力随 n 增大被稀释（远距离 token 分到的权重变小）+ 训练语料长程依赖样本稀少 + 位置编码远端区分度衰减 + 因果注意力让首部成锚点、尾部有近因优势，夹在中间的最吃亏。综合表现为 U 型召回曲线。

### Q5：FlashAttention 扩展了上下文长度吗？
没有改 O(n²) 的算力复杂度，也不是近似。它通过分块 + online softmax **不实体化完整 n×n 注意力矩阵**，把注意力显存从 O(n²) 降到 O(n)，并大幅减少 HBM 读写。效果是同样硬件能跑更长序列、更快——它让长上下文"可行"，但本质仍是精确的平方阶注意力。

### Q6：MQA / GQA 解决的是什么问题？
KV cache 显存。多头注意力里每个 query 头都有独立 KV 头，KV cache 很大；MQA 让所有 query 头共享 1 组 KV，GQA 折中（分组共享）。代价是轻微质量损失，换来 KV 显存和带宽的大幅下降，是长上下文推理的标配。

### Q7：prompt caching 和 KV cache 是什么关系？
prompt caching 就是把稳定前缀（system + tools + 早期历史）已经算好的 **KV 缓存复用**，命中后这段不再重新做前向注意力计算，只为新增的增量付费。因为 agent loop 每轮前缀高度重叠，所以 prompt caching 在 loop 场景收益极大——它直接利用了 KV cache 可复用这个事实。

## 九、关键术语 Cheat Sheet

| 术语 | 一句话 |
|------|--------|
| 上下文窗口 | 一次前向能参与注意力计算的最大 token 数 |
| 自注意力 O(n²) | 每 token 对所有 token 算分，n×n 矩阵，成本平方阶 |
| KV Cache | 缓存每层每 token 的 K/V，推理期显存瓶颈，随 n 线性 |
| MQA / GQA | query 头共享少量 KV 头，压 KV cache 显存 |
| RoPE | 旋转位置编码，相对位置，可有限外推 |
| ALiBi | 注意力距离线性偏置，外推友好 |
| 位置插值 / YaRN | 长度外推：把超长位置压回训练范围 |
| Lost in the Middle | 中段信息召回率低，U 型曲线 |
| FlashAttention | IO 感知精确注意力，显存 O(n²)→O(n)，不改算力阶 |
| StreamingLLM | 保留 sink token 支撑流式无限上下文 |
| Mamba / RWKV | 线性/SSM，用状态替代全注意力，O(n) 推理 |
| Prompt Caching | 复用稳定前缀的 KV cache，loop 场景大幅省成本 |

## 十一、核心要点速记

1. "上下文窗口问题"分两层：硬上限（能装多少）+ 软退化（装了能不能用好）。
2. 自注意力是 O(n²)，n 翻倍成本翻四倍——长上下文贵的训练期根因。
3. 推理期真正的瓶颈是 KV cache 显存，随 token 数 × 层数 × 头数线性增长。
4. 位置编码决定外推能力；标称长窗口常靠 RoPE 外推硬撑，末端质量打折。
5. Lost in the Middle：注意力稀释 + 训练偏置 + 位置衰减，让中段成洼地。
6. 扩展上下文都是成本↔精度的权衡；FlashAttention 是少数纯赚（省显存不改算力阶）。
7. 上下文工程是对抗这些架构硬约束的必然；prompt caching 本质是复用 KV cache。
