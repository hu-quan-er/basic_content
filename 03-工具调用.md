# 工具调用（Tool Use / Function Calling）

> 目标：掌握 Function Calling 的工程细节、并行调用与工具依赖的处理、Function Calling 的局限与"代码 workflow 替代"的判断，以及工具治理、错误处理、沙箱执行。
> MCP 协议单独拆到《[03.1-MCP](./03.1-MCP.md)》。

---

## 一、核心概念

### 1.1 工具（Tool）的本质
- **Tool = LLM 的"手"**：模型只产出意图（JSON），实际执行由宿主程序完成。
- **三层契约**：
  - **Schema**（机器接口）：name、description、parameters JSON Schema
  - **描述**（给 LLM 看的接口）：自然语言指引模型何时、如何调用
  - **实现**（实际执行）：宿主代码 / API / RPC

### 1.2 工具调用的完整生命周期
```
1. Schema 注入   → 把所有可用工具的 schema 加入 system prompt
2. 模型决策      → LLM 输出 tool_call(name, arguments)
3. 调用解析      → 宿主程序解析 JSON、校验 schema
4. 实际执行      → 调用真实函数 / API
5. 结果回填      → 把结果作为 tool_result message 传回
6. 继续推理      → LLM 基于结果继续生成（可能再调工具，或最终回答）
```
这其实就是 agent loop 的一轮（详见《Agent Loop专题》）。

### 1.3 工具描述的最佳实践（重点）
- **name**：动词开头、snake_case、唯一明确（`search_orders` 而非 `query`）。
- **description**：
  - 第一句讲清楚"做什么"
  - 第二句讲清楚"何时调用"（边界）
  - 列出反例（"不要用此工具查 X"）
- **参数**：
  - 类型严格（Enum 优于 string）
  - description 字段必填，给出格式示例
  - 必填 vs 可选要明确
```json
{
  "name": "search_orders",
  "description": "Search user orders by date range. Use this when user asks about purchase history, refunds, or shipping status. Do NOT use for product inventory queries (use search_products instead).",
  "parameters": {
    "user_id": {"type": "string", "description": "User UUID, e.g., 'usr_abc123'"},
    "start_date": {"type": "string", "format": "date", "description": "YYYY-MM-DD"},
    "status": {"type": "string", "enum": ["pending", "shipped", "delivered", "cancelled"]}
  }
}
```

### 1.4 工具选择的三大失败模式
1. **描述模糊**：两个工具描述重合 → 模型选错。
2. **参数歧义**：模型猜参数（如把"昨天"猜成 `date="yesterday"` 字符串）。
3. **工具过多**：超过 20+ 工具，模型 attention 稀释，选择质量明显下降。

## 二、并行调用与工具依赖（深入）

### 2.1 并行工具调用是怎么工作的

- **模型侧**：一轮里模型可以**一次输出多个 tool_call**（OpenAI 多个 `tool_calls`、Anthropic 多个 `tool_use` block）。
- **宿主侧**：宿主拿到这一批 tool_call 后**并发执行**（`asyncio.gather` / `Promise.all` / Go 的 goroutine + WaitGroup），全部完成后把多个 tool_result 一起回填，模型继续。
- **收益**：N 个独立调用的延迟从 `N × 单次` 降到 `max(单次)`。

```python
# 宿主侧并发执行同一轮的多个 tool_call
results = await asyncio.gather(*[run_tool(c) for c in resp.tool_calls],
                               return_exceptions=True)
```

适用：相互独立的查询——"查北京和上海的天气""拉这 5 个仓库的 star 数"。

### 2.2 工具之间有依赖怎么办？

**关键认知：有依赖的调用不能放在同一批并行里**——因为 B 的参数依赖 A 的返回，而并行时 A 还没出结果。处理方式分两种：

#### 方式一：靠 loop 多轮串行（FC 的默认方式）
依赖关系**不是在 API 里"配置"出来的，而是 loop 天然表达的**：
```
第 1 轮：模型调用 A（拿到 order_id 之前不可能调 B）
   ↓ 宿主执行 A，回填结果
第 2 轮：模型看到 A 的结果，再调用 B(order_id=A.result)
```
模型"看到上一步结果再决定下一步"，依赖就被串行的轮次自然解决了。**你要做的不是配置依赖图，而是把每个工具的描述写清楚**（B 的描述里说明"需要先有 order_id"），让模型知道先后。

#### 方式二：把依赖显式建成 DAG（代码编排）
当依赖关系**固定且已知**时，与其每轮问模型，不如直接在代码里把依赖写成 DAG，前置节点跑完把结果喂给后置节点——这就是第四节要讲的"用 workflow 替代 FC"。

#### 一轮里混合"并行 + 依赖"
现实常是：a1、a2 互相独立（可并行），但 b 依赖二者。模型的正确行为是：
```
第 1 轮：并行 a1、a2          ← 同批并行
第 2 轮：拿到 a1、a2 结果后调 b ← 下一轮串行
```
**模型并不总能正确切分**，所以要：① 工具描述讲清依赖；② 必要时用 `tool_choice` 引导；③ 关键依赖直接上代码编排，别赌模型。

### 2.3 控制并行行为（provider 层）
- **OpenAI**：`parallel_tool_calls=true/false`，可关掉强制每轮最多一个工具（依赖强、要严格串行时关掉更稳）。
- **Anthropic**：默认可并行，可在 prompt 里要求"一次只调一个"。
- **结构化输出 / `tool_choice`**：可强制本轮必须调某工具或必须调工具，用来纠正"该并行不并行 / 该串行乱并行"。

### 2.4 并行的坑
1. **隐性依赖**：模型误判为可并行，实际 B 依赖 A → 结果错乱。对策：描述写清依赖、强依赖关并行。
2. **副作用 / 竞态**：写操作并行可能并发改同一资源。对策：只读/幂等工具才标 `parallel_safe`，写操作默认串行。
3. **资源限制**：并行打爆 API rate limit / DB 连接池。对策：并发上限 + 限流。
4. **错误处理复杂**：5 个并行 1 个失败怎么办。对策：`return_exceptions=True` 收集，部分成功也能继续。

## 三、Function Calling 一定就是好的吗？

**不一定。** Function Calling 的本质是"让模型在运行时决定调什么"，它的灵活性是有代价的：

### 3.1 Function Calling 的代价
| 代价 | 说明 |
|------|------|
| **每步一次 LLM 往返** | N 个工具 ≈ N+1 次模型调用，延迟和成本线性甚至平方累积（历史增长，见《Agent Loop专题》第十节） |
| **非确定性** | 模型可能选错工具、猜错参数、漏掉步骤、该停不停 |
| **难审计 / 难保证** | 无法保证它每次都按同一固定顺序执行——合规、可复现场景是硬伤 |
| **对固定流程是浪费** | 流程已知还每轮问模型"下一步做什么"，既慢又不稳 |

### 3.2 什么时候 FC 是对的，什么时候不是
- **适合 FC（动态决策）**：下一步依赖中间结果、分支多且不可预知、开放式任务（"帮我调研并写报告"）。
- **不适合 FC（路径已知）**：固定的数据处理流水线、ETL、"先查 A 再查 B 再算 C 再渲染"这种**步骤和依赖都确定**的场景——这时让模型来编排是负优化。

**核心结论**：FC 解决的是"不知道下一步该干什么"的问题。如果你**已经知道**步骤和依赖，就不该用 FC 去"假装让模型决定"，而应该用代码把流程固定下来。

## 四、能不能用代码 Workflow 替代 Function Calling？

**能，而且路径已知时往往更好。** 这正是 Anthropic《Building effective agents》的核心建议：**用能解决问题的最简方案，Workflow 优先于 Agent**（见《LLM基础与Agent本质》Workflow vs Agent）。

### 4.1 核心思路：路径已知就别让模型编排
如果"调用哪些工具、什么顺序、谁依赖谁"是确定的，就在**代码里把它写成固定流程**，前置工具跑完，把结果直接塞进上下文，让模型只做它真正擅长的那部分（理解、归纳、生成）。

### 4.2 "前置跑完把结果塞进去"（Prefetch / Context Stuffing）
这是你问的那个模式，非常实用：

```python
# 反例：让模型自己决定调这三个工具（3+ 次 LLM 往返、可能调错）
#   → 用 function calling 跑一个其实固定的流程

# 正例：代码确定性地前置执行，把结果塞进 prompt，模型一次推理
profile  = get_user_profile(user_id)        # 已知一定要查
orders   = get_recent_orders(user_id)       # 已知一定要查
coupons  = get_available_coupons(user_id)   # 已知一定要查

answer = llm(f"""用户资料：{profile}
最近订单：{orders}
可用优惠券：{coupons}
请基于以上信息回答用户问题：{question}""")     # 一次调用，确定、便宜、快
```

- **优点**：确定（一定会查、不会漏/错）、省（少几次 LLM 往返）、快（前置可并行）、可审计（流程在代码里）。
- **代价**：失去动态性——如果某些场景其实不需要全部前置数据，会做无用功。

### 4.3 Workflow vs Function Calling 对比
| 维度 | 代码 Workflow（前置编排） | Function Calling（模型编排） |
|------|---------------------------|------------------------------|
| 确定性 | 高，固定流程 | 低，模型每轮决定 |
| 延迟/成本 | 低（少 LLM 往返，前置可并行） | 高（每步一次往返） |
| 灵活性 | 低，路径写死 | 高，动态分支 |
| 可审计/可复现 | 强 | 弱 |
| 适用 | 步骤/依赖**已知固定** | 步骤**不可预知** |

### 4.4 混合模式（生产最常用）
**确定性骨架 + 关键不确定点才 agentic**：
```
[代码 workflow]
  ├─ 前置并行：查资料 / 查订单 / 查库存   ← 固定，代码跑
  ├─ 模型决策点：根据情况选择处理分支       ← 不确定，这里才用 FC/agent
  └─ 后置：渲染 / 落库 / 通知              ← 固定，代码跑
```
把 LLM 的"自由裁量"收窄到真正需要它的那一两个点，其余交给确定性代码。这通常是延迟、成本、可靠性的最优解。

### 4.5 决策矩阵：到底用哪个
```
这一步的"下一步动作"是否可以提前确定？
├─ 是 → 代码 workflow（前置跑完塞结果），不要用 FC
└─ 否
     是否需要多轮、依赖中间结果动态决策？
     ├─ 是 → Function Calling / agent loop
     └─ 介于之间 → 混合：确定性骨架 + 关键点 agentic
```

**一句话**：FC 不是越多越好。先问"这步能不能提前确定"，能确定就用代码，把 FC 留给真正不确定的决策点。

## 五、工具治理：工具一多怎么办

### 5.1 Tool Retrieval（工具检索 / Tool RAG）
**问题**：工具数 > 30 时全量注入 prompt → token 浪费 + 选择准确率下降 + Lost in the Middle。
**解法**：把工具描述 embedding 化，按用户问题召回 top-k（通常 5~10），只把这 k 个注入。
**要点**：每个工具一条 embedding（name + description）；BM25 + 向量混合检索效果最好；召回后可二次确认再注入。

### 5.2 三种治理路线（按规模选）
| 工具规模 | 路线 |
|----------|------|
| < 20 | 全量注入即可 |
| 20~50 | 开始上 Tool Retrieval |
| 50+ | 必须 retrieval 或分层 |
| 100+ | 分层 Agent + retrieval 组合 |

- **Tool Retrieval**：向量召回 top-k 再注入。
- **分层 Agent**：Router 选子领域 → 子 Agent 只看本域工具（5~10 个）。
- **命名空间 + 渐进披露**：工具按 `orders.search` 组织，先选 namespace 再展开（也是 MCP 的方向，每个 server 一个 namespace）。

## 六、工具结果与错误处理

### 6.1 结果处理
- **成功**：回填 JSON 结果。
- **大结果**：截断 + 摘要，或引用化（存 artifact 返回 id），不要把 10MB 原样喂模型（详见《16.2-Agent Loop上下文管理》）。

### 6.2 错误三分类
| 错误类型 | 处理 |
|----------|------|
| 临时性（网络、超时、429） | 宿主层指数退避重试（~3 次），**模型不感知** |
| 业务错误（参数错、404、422） | 结构化错误回填让模型自纠；同参数失败 ≥2 次强制跳出 |
| 致命错误（权限、配额） | 立即终止，返回友好错误 + 告警 |

```python
TOOL_RETRY_POLICY = {
  "max_total_calls": 20,   # 全程工具调用上限
  "max_same_tool": 5,      # 同一工具上限
  "max_same_args": 2,      # 同参数重试上限
  "transient_retry": 3,    # 临时错误重试
  "timeout_per_call": 30,  # 单次超时
}
```
（重试与副作用冲突、幂等设计见《Agent工程细节与可靠执行》。）

## 七、沙箱执行（Code Execution）

当 Agent 需要执行代码（Code Interpreter）时，安全是第一位的。

**风险**：任意代码执行、逃逸宿主、数据泄露、资源滥用、网络滥用。
**七层防御**：
1. **隔离运行时**：Docker（基础）/ Firecracker、gVisor microVM（更强）/ 一次性容器。
2. **配额**：CPU 1 core、内存 512MB、超时 30s、磁盘只读根 + /tmp 限额。
3. **网络**：默认无网，必要时白名单，内网禁访。
4. **文件系统**：只读根，可写仅 /tmp，上传文件挂载只读。
5. **包管理**：预装常用包，禁任意 pip install（或 PyPI 白名单）。
6. **审计监控**：全量日志（源码/stdout/stderr）+ 异常告警。
7. **托管方案**：E2B、Modal、Daytona。

## 九、思考题（含解答）

### Q1：工具描述怎么写才能让模型准确调用？
四要素：①第一句讲做什么（动词开头）；②讲何时调用（边界）；③讲何时**不要**调用（反例，很重要）；④每个参数都给 description + 格式/单位/示例。进阶：system prompt 里给 1~2 个 few-shot 调用示例、用 Enum 收紧参数、主动构造歧义场景测试。评估：建"用户问题→期望调用"测试集，改 description 就回归。

### Q2：工具之间有依赖时，并行调用怎么处理？
有依赖的不能放同一批并行（B 的参数依赖 A 的返回，并行时 A 还没出结果）。两种解法：①靠 loop 多轮串行——模型这轮调 A、看到结果下轮再调 B，依赖由轮次自然表达，你要做的是把工具描述里的先后讲清楚；②依赖固定已知时，直接在代码里建成 DAG，前置跑完喂后置（即用 workflow 替代 FC）。一轮里"独立的并行、依赖的留到下轮"，模型未必切分对，强依赖建议关并行或上代码编排。

### Q3：Function Calling 一定比代码编排好吗？
不是。FC 的代价是每步一次 LLM 往返（慢、贵）、非确定（可能选错/漏步）、难审计。它适合"下一步不可预知"的动态决策；对步骤和依赖**已知固定**的流程，用 FC 让模型编排是负优化。判断标准：这步的下一步动作能否提前确定——能确定就用代码。

### Q4：什么是"前置跑完把结果塞进去"，好处是什么？
就是把已知一定要调的工具在代码里确定性地（可并行）先跑完，把结果直接拼进 prompt，让模型只做一次推理，而不是用 function calling 让模型一轮轮去调。好处：确定（不漏不错）、省（少 LLM 往返）、快（前置并行）、可审计（流程在代码里）。代价是失去动态性，可能对某些不需要全部数据的场景做无用功。生产里常用混合：确定性骨架 + 关键不确定点才 agentic。

### Q5：50+ 工具的 Agent 怎么治理？
全量注入行不通。三条路线：①Tool Retrieval——工具描述 embedding 化，按问题召回 top-10 再注入（BM25+向量混合最好）；②分层 Agent——Router 选域、子 Agent 只看本域工具；③命名空间+渐进披露——先选 namespace 再展开。规模经验：<20 全量、20~50 retrieval、50+ 必须 retrieval/分层、100+ 组合。

### Q6：工具调用失败如何兜底？
三分类：①临时错误（网络/超时/429）宿主层指数退避重试、模型不感知；②业务错误（参数错/404/422）结构化错误回填让模型自纠，同参数失败≥2 次强制跳出；③致命错误（权限/配额）立即终止+告警。整体加全程调用上限、同工具上限、同参数上限、单次超时。全部失败则降级返回"已记录工单"+人工接管。

### Q7：Code Interpreter 的安全风险怎么防？
风险：任意代码执行、逃逸、数据泄露、资源/网络滥用。七层防御：隔离运行时（Docker/microVM/一次性容器）、配额（CPU/内存/超时/磁盘）、网络（默认无网+白名单）、文件系统（只读根+/tmp）、包管理（预装+禁任意安装）、审计监控（全量日志+告警）、用托管沙箱（E2B/Modal/Daytona）。

## 十、场景设计题（含解答）

### 场景 1：企业接入 200+ 内部 API
**架构**：Agent Orchestrator(LangGraph) → Tool Gateway（Registry + 向量召回 top-15 + 权限过滤 + 审计 + 限流）→ 按业务域分的工具组（crm/erp/hr/finance）→ 真实 API。
**关键决策**：①OpenAPI 自动生成 schema + 人工润色高频 API（top 20% 占 80% 调用）；②召回 top-15 按 namespace 分组注入；③工具标 `required_roles`，用户 SSO 后过滤，敏感操作强制 HITL；④全链路 trace + 离线分析误调用；⑤分期演进（20→100→200→引入 Tool RAG）。
**注意**：对固定的多步流程（如"取数→校验→落库"）直接用代码 workflow，不要都塞给模型编排。

### 场景 2：金融数据分析 Agent（查财报→算指标→出图）
**工具集**：数据获取（`get_company_financials`、`get_stock_price`、`get_news`）、计算（复杂计算交 **Code Interpreter** 而非穷举写工具）、可视化（`generate_chart`）、辅助（`current_date` 防幻觉）。
**关键决策**：①固定分析流程用代码编排 + 前置并行取数，把 DataFrame 塞给 Code Interpreter；②数据工具统一返回序列化 DataFrame；③财务数据确定性强→加缓存；④大结果工具内摘要后返回。
**安全**：Code Interpreter 跑在 E2B 沙箱，只允许预装数据科学包，图表存对象存储返回 URL。
