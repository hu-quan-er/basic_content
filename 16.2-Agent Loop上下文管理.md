# Agent Loop 上下文管理（子文档）

> 母文档：《[16-Agent Loop 专题](./16-Agent%20Loop专题.md)》第五节。
> 这里把"loop 里历史单调增长怎么管"讲透：五种控制策略逐一详解、组合 pipeline、可观测评估，以及 LangChain / LangGraph / LlamaIndex / Claude Agent SDK / OpenAI Assistants / MemGPT(Letta) / Mem0 等主流实现的对比分析。
> 为什么会有窗口上限/软退化的**模型原理**，见《[17-上下文窗口的模型原理](./17-上下文窗口的模型原理.md)》——本篇是它的工程落地。

## 一、问题定位

agent loop 每轮新增 assistant 的 thought + tool_call + 每个 tool 的返回，**历史单调增长**，工具返回（网页、日志、JSON）往往是大头。两个压力：

- **硬上限**：撞 context window（O(n²) 算力 + KV cache 显存，见 17 号文档）→ 撞了就崩或被迫截断。
- **软退化 + 成本**：即使没满，过长上下文也会 Lost in the Middle、且成本随轮数近似平方增长。

所以上下文管理不是"能不能装下"，而是**在每一轮主动决定"留什么、压什么、丢什么、外置什么"**。

**核心判断**：上下文管理 = 在「信息完整性 ↔ token 成本 ↔ 注意力质量 ↔ 延迟」之间做每轮调度。没有最优解，只有针对任务的策略组合。

---

## 二、目标函数：什么是"管好了"

| 维度 | 想要 | 冲突点 |
|------|------|--------|
| 信息完整性 | 决策需要的信息都在 | 与成本/质量冲突 |
| token 成本 | 每轮尽量短 | 与完整性冲突 |
| 注意力质量 | 关键信息在模型"看得清"的位置 | 与完整性冲突（塞太多反而稀释） |
| 延迟 | 上下文短、可缓存 | 与完整性冲突 |
| 可恢复 | 关键状态不因压缩丢失 | 压缩不可逆 |

**一句话目标**：每轮只保留"决策这一步所必需的最小充分信息"，其余的压缩、外置或丢弃，且关键事实先固化再压缩。

## 三、五种控制策略详解

### 3.1 截断 / 窗口（Truncation / Windowing）

- **做法**：只保留最近 N 轮 / 最近 K 个 token，超出的从最早开始丢。
- **变体**：
  - 固定轮数窗口（保留最近 N 条消息）。
  - token 预算窗口（保留到 token 上限）。
  - **首尾保留**（keep head + tail）：保留最早的系统设定/目标 + 最近若干轮，丢中段——直接对抗 Lost in the Middle。
- **优点**：实现最简单、零额外 LLM 调用、延迟低、对 prompt caching 友好（前缀稳定）。
- **缺点**：硬丢信息，早期关键事实会被丢掉。
- **适用**：短程任务、闲聊、对历史依赖弱的场景。
- **主流实现**：LangChain `ConversationBufferWindowMemory(k)`、`ConversationTokenBufferMemory`；LangGraph `trim_messages(strategy="last")`；OpenAI Assistants `truncation_strategy={"type":"last_messages"}`。

### 3.2 摘要 / 压缩（Summarization / Compaction）

- **做法**：把旧历史用一次 LLM 调用压成摘要，用摘要替换原文继续。
- **变体**：
  - **滚动摘要（running summary）**：维护一份不断更新的对话摘要。
  - **递归/分段摘要**：长内容分块各自摘要再合并。
  - **触发式压缩**：达阈值（如 70% 窗口）才压，而非每轮压。
  - **混合（recent verbatim + old summary）**：近 N 轮原文 + 更早的摘要，兼顾细节与长度。
- **优点**：保留语义、长程任务可持续推进。
- **缺点**：摘要会丢细节、有额外 LLM 调用成本与延迟、**不可逆**（压错了找不回）、破坏 prompt cache 前缀。
- **工程要点**：保护区（system / 当前目标 / 最近一轮永不压）；压缩前把关键事实固化到结构化 state；记录"已压缩到第几轮"便于审计。
- **适用**：长对话、长 agent 任务、deep research。
- **主流实现**：LangChain `ConversationSummaryMemory` / `ConversationSummaryBufferMemory`（混合）；LangGraph + `langmem` 的 `SummarizationNode`；LlamaIndex `ChatSummaryMemoryBuffer`；**Claude Agent SDK / Claude Code 的 auto-compaction**；ChatGPT 长对话自动摘要。

### 3.3 外置记忆（Scratchpad / External Memory）

- **做法**：把关键结论、变量、事实写到上下文**之外**的存储（结构化 state / KV / 向量库 / 文件），context 里只留指针或按需检索回来。
- **两类**：
  - **Scratchpad（工作记忆）**：本 run 内的中间变量、已知事实，结构化存，loop 每轮可读写，结束即弃。
  - **长期记忆**：跨 run 的知识/偏好，存数据库/向量库，按需检索注入。
- **优点**：context 不随信息量线性膨胀、关键信息不怕被压缩丢失、可跨 run 复用。
- **缺点**：检索可能漏（召回错就等于没记）、引入额外系统复杂度。
- **适用**：信息量大、需跨轮/跨会话稳定记忆的任务。
- **主流实现**：MemGPT/Letta 的 `core_memory` / `archival_memory`；Mem0 事实记忆层；LangGraph state + `langmem` 记忆工具；LlamaIndex `VectorMemory` / Memory blocks；**Claude 的 memory tool（模型自管文件记忆）**。

### 3.4 分层记忆（Hierarchical Memory）

- **做法**：仿操作系统的虚拟内存——**主上下文（in-window）+ 外部上下文（out-of-window）**，由调度器在两者间换入换出（paging）。模型用函数自己决定把什么调进来、把什么写出去。
- **典型分层**：工作记忆（当前）/ 情节记忆（近期对话）/ 长期记忆（持久知识）。
- **优点**：理论上支持"无限"上下文、模型可主动管理自己的记忆。
- **缺点**：实现最复杂、换入换出的调度策略本身要调优、延迟更高。
- **适用**：长生命周期 agent、个人助理、需要"记住用户"的场景。
- **主流实现**：**MemGPT / Letta**（最彻底的分层分页 + self-editing memory）；部分平台的"记忆"功能（ChatGPT Memory）思想相近。

### 3.5 工具结果引用化（Artifact / Reference）

- **做法**：大工具结果（8MB 日志、长网页、大 JSON）不原样回填，存成 artifact 返回一个 `id` + 简短摘要；模型要细节时再用 `fetch(id, query)` 取片段。
- **优点**：单个大结果不再撑爆 context、按需取用、可对结果做二次检索。
- **缺点**：多一次"取数"往返、需要 artifact 存储。
- **适用**：工具返回体积大、但每轮只用其中一小部分的场景（日志分析、长文档、爬取）。
- **主流实现**：多数 coding agent 对文件/检索结果的句柄化处理；RAG over tool output；自建 artifact store。

## 四、组合策略：生产里从来不是单选

真实系统通常**多种叠加**成一条 pipeline。一个典型的 agent loop 上下文管理栈：

```
稳定前缀（system + tools）          → prompt caching 复用 KV
   +
最近 N 轮原文（recent verbatim）    → 保证近因信息清晰
   +
更早历史 → 触发式摘要（≥70% 才压） → 控制总长，保护区不动
   +
关键事实/结论 → 固化到结构化 state  → 压缩不丢、可恢复
   +
大工具结果 → artifact 引用化        → 单条不撑爆
   +
跨 run 知识 → 长期记忆按需检索       → 不常驻 context
```

**设计顺序**：先用"稳定前缀 + 窗口"拿到缓存与低延迟的底子，再叠"触发式摘要"扛长度，再用"state 固化 + 引用化"防关键信息丢失，最后用"长期记忆"做跨会话。

## 五、主流实现对比分析（核心）

### 5.1 逐家速览

| 实现 | 主打策略 | 关键机制 / API | 自动化 | 特点 |
|------|----------|----------------|--------|------|
| **LangChain memory（经典）** | 窗口 / 摘要 / 混合 | `BufferWindowMemory` / `SummaryMemory` / `SummaryBufferMemory` / `TokenBufferMemory` | 半自动 | 开箱即用，正逐步被 LangGraph 持久化取代 |
| **LangGraph** | 截断 + 摘要 + 删消息 | `trim_messages` / `RemoveMessage` / `pre_model_hook` / `langmem` | 自定义 | 上下文管理完全可编程、可放进图节点 |
| **LlamaIndex** | token buffer + 摘要 + 向量 | `ChatMemoryBuffer` / `ChatSummaryMemoryBuffer` / `VectorMemory` / Memory blocks | 半自动 | 记忆"块"可组合（事实/静态/向量） |
| **Claude Agent SDK / Code** | 自动压缩 + 上下文编辑 + 记忆工具 | auto-compaction、context editing（clear 旧 tool 结果）、memory tool（文件记忆） | 自动 | 产品级开箱，逼近窗口时自动接管 |
| **OpenAI Assistants** | 截断策略 | `truncation_strategy`: `auto` / `last_messages` | 自动 | 简单，粒度较粗 |
| **MemGPT / Letta** | 分层分页 | main / external context、self-editing memory（`core_memory_*`、`archival_*`） | 自动 | OS 类比，最彻底的分层虚拟上下文 |
| **Mem0** | 外置事实记忆 | `add` / `search` 事实抽取与检索层 | 半自动 | 专注跨会话长期记忆，可接到任意框架 |

### 5.2 对比分析（看门道）

1. **"框架托管 vs 产品自动"两条路线**：
   - LangChain/LangGraph/LlamaIndex 把策略**做成可拼装的组件**，控制权在开发者（适合要定制的系统）。
   - Claude Agent SDK / OpenAI Assistants 把上下文管理**藏进产品**，逼近窗口自动压缩/截断（适合快速落地，但黑盒、不可细控）。

2. **"压缩派 vs 外置派"的根本分歧**：
   - 压缩派（Summary 系、Claude compaction）：信息**留在上下文里**但变短，简单、连续，但有损、不可逆。
   - 外置派（MemGPT/Letta、Mem0、向量记忆）：信息**搬出上下文**按需取回，可无限扩展、可跨会话，但依赖检索质量。
   - 趋势：**两者结合**——近期压缩 + 关键事实外置。

3. **MemGPT 的独特价值**：把"上下文管理"从开发者手里交还给**模型自己**（self-editing memory），最接近"无限上下文"的理念，但调度开销与复杂度也最高。

4. **截断仍是底座**：再花哨的方案，最近 N 轮原文 + 稳定前缀缓存几乎都是标配——因为它便宜、低延迟、对 prompt cache 友好。

5. **谁动了 prompt cache**：摘要/压缩会改写前缀、**打破缓存命中**；窗口/引用化/外置记忆对缓存更友好。生产里要权衡"压缩省的 token" vs "破缓存多花的钱"。

### 5.3 选型一句话

- 要**快速上线、不想操心** → Claude Agent SDK / OpenAI Assistants 的自动管理。
- 要**精细可控、自定义控制流** → LangGraph（trim + 摘要节点 + state）。
- 要**跨会话长期记忆** → Mem0 / MemGPT(Letta) / 向量记忆，叠到上面任意一种。
- 要**省钱保缓存** → 稳定前缀 + 窗口 + 引用化为主，摘要为辅。

## 六、选型决策树

```
任务有多长？
├─ 短（几轮、弱历史依赖）        → 截断/窗口即可
└─ 长（多轮/长任务）
     需要跨会话记住用户/知识吗？
     ├─ 否 → 触发式摘要 + 关键事实 state 固化（+ 大结果引用化）
     └─ 是 → 上面的基础上叠 长期记忆（Mem0/向量）或 分层记忆（MemGPT）
     成本/延迟敏感、要保 prompt cache？
     ├─ 是 → 少用全量摘要，多用窗口 + 引用化 + 外置
     └─ 否 → 可上混合摘要换更好连续性
```

## 七、可观测与评估：怎么知道"管得好不好"

上下文管理是**隐性**的，必须可观测才能调优：

- **每轮记录**：context token 数、压缩触发次数、被丢/被压的内容、缓存命中率、本轮成本。
- **质量信号**：是否出现"忘了先前约束""重复检索已有信息""前后矛盾"——这些是上下文管理失败的征兆。
- **回归评测**：构造长程任务测试集，改压缩策略后跑评测，确认没掉准确率（防"压过头丢关键信息"）。
- **关键指标**：平均 context 长度、缓存命中率、单任务成本、长程任务完成率。

**核心结论**：上下文管理不可"拍脑袋"，要靠逐轮 trace + 长程评测集来验证策略有没有把关键信息压没（详见《评估与可观测性》）。

## 八、思考题（含解答）

### Q1：为什么生产系统很少只用一种上下文策略？
因为各策略在"信息完整性/成本/质量/缓存友好/可恢复"上各有取舍，单一策略必然在某个维度吃亏。典型做法是叠成 pipeline：稳定前缀缓存 + 最近 N 轮原文 + 触发式摘要 + 关键事实 state 固化 + 大结果引用化 + 长期记忆按需检索，各补一段。

### Q2：摘要压缩有什么隐藏代价？
①有损且不可逆，压错信息找不回；②额外一次 LLM 调用的成本与延迟；③**会改写前缀、打破 prompt cache 命中**，可能"省了 token 却多花了钱"。所以要触发式（达阈值才压）+ 保护区 + 压缩前把关键事实固化到结构化 state。

### Q3：MemGPT 的分层记忆和普通"摘要+向量记忆"本质区别在哪？
普通方案由**开发者**预设何时摘要、检索什么；MemGPT 把记忆管理**交给模型自己**——模型通过函数调用主动把信息在主上下文与外部上下文之间换入换出（self-editing），类比操作系统的虚拟内存分页。更接近"无限上下文"，但调度开销和复杂度最高。

### Q4：为什么说"截断/窗口"虽然最简单却淘汰不掉？
因为它零额外 LLM 调用、延迟最低，且**保持前缀稳定、对 prompt caching 最友好**。再复杂的方案通常也以"稳定前缀 + 最近 N 轮原文"为底座，只是在更早的历史上叠摘要/外置。

### Q5：Claude Agent SDK 的自动压缩和 LangGraph 的手动管理，分别适合谁？
自动压缩（逼近窗口时产品接管）适合想**快速落地、不愿操心**的场景，代价是黑盒、不可细控。LangGraph 把 trim/摘要/删消息做成可编程节点，适合需要**精细控制、自定义触发与保护区**的生产系统，代价是要自己设计。

## 九、关键术语 Cheat Sheet

| 术语 | 一句话 |
|------|--------|
| Truncation / Window | 只保留最近 N 轮/token，最简单、缓存友好 |
| Keep head+tail | 保留首尾丢中段，对抗 Lost in the Middle |
| Compaction / Summary | 把旧历史压成摘要，有损、不可逆、破缓存 |
| Running summary | 不断更新的对话摘要 |
| Recent verbatim + old summary | 近 N 轮原文 + 更早摘要的混合策略 |
| Scratchpad | 本 run 内的工作记忆，结构化外置 |
| 长期记忆 | 跨 run 知识/偏好，按需检索注入 |
| Hierarchical memory | 主上下文+外部上下文分页（MemGPT） |
| Self-editing memory | 模型自己读写记忆（MemGPT/Letta、Claude memory tool） |
| Artifact 引用化 | 大结果存外部返回 id+摘要，按需取 |
| 触发式压缩 | 达阈值（如 70%）才压，而非每轮 |
| 保护区 | system/目标/最近一轮永不压 |

## 十一、核心要点速记

1. 上下文管理 = 每轮在「完整性/成本/注意力质量/延迟/可恢复」间做调度，无最优解只有组合。
2. 五种策略：截断窗口、摘要压缩、外置记忆、分层记忆、结果引用化。
3. 生产是组合：稳定前缀缓存 + 近 N 轮原文 + 触发式摘要 + state 固化 + 引用化 + 长期记忆。
4. 两条路线：框架托管（LangChain/LangGraph/LlamaIndex 可拼装）vs 产品自动（Claude/OpenAI 黑盒接管）。
5. 两派分歧：压缩派（留在上下文变短，有损）vs 外置派（搬出去按需取，靠检索）；趋势是结合。
6. 截断/窗口淘汰不掉：便宜、低延迟、对 prompt cache 最友好，是所有方案的底座。
7. 摘要会破 prompt cache，要权衡"省的 token vs 破缓存多花的钱"。
8. 上下文管理要可观测 + 长程评测，确认没把关键信息压没。
