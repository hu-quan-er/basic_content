# Agent Loop 停止条件设计（子文档）

> 母文档：《[16-Agent Loop 专题](./16-Agent%20Loop专题.md)》第四节。
> 这里把"loop 怎么停"单独讲透：设计维度、完成信号陷阱、死循环检测、收尾策略，以及 LangChain / LangGraph / OpenAI Agents SDK / Claude / AutoGen / CrewAI 等主流产品/框架的真实停止机制，配 Python 与 Go（[eino](https://github.com/cloudwego/eino)）代码。

---

## 一、为什么停止条件最容易出事故

agent loop 的停止逻辑是"两难"：

- **停不下来**：死循环、反复调同一工具、刷爆 API 配额、烧钱、把上下文撑爆后草率收尾。
- **停太早**：任务没真正完成就 finish，或触顶预算被强制中断却没标注"未完成"，把半成品当成品交付。

单次 prompting 不存在这个问题（一问一答天然结束），是 agent loop 独有的工程难点。**核心目标：在"任务确实做完了"和"系统失控"之间，用多道显式闸卡住循环。**

---

## 二、停止条件的两层

| 层 | 含义 | 期望 |
|----|------|------|
| **正常终止（success）** | 任务真正完成，模型主动收尾 | 越精确越好，避免假完成 |
| **兜底终止（safety）** | 预算/循环/错误/外部触发的强制停止 | 越早发现越好，必须标注未完成 |

一个健壮的 loop = 1 个精确的完成信号 + N 道兜底闸。**只有完成信号没有兜底闸 = 会失控；只有兜底闸没有精确完成信号 = 会假完成或浪费预算。**

---

## 三、五个设计维度

把所有停止条件归纳为五类，生产系统通常五类都要有：

| 维度 | 触发信号 | 典型参数 | 处置 |
|------|----------|----------|------|
| **完成型** | 模型表示任务做完 | finish 工具 / 无 tool_call / stop sequence / 结构化 final_output | 返回结果 |
| **预算型** | 资源用尽 | max_steps、max_turns、max_tokens、max_cost、max_wall_time、max_rpm | 强制收尾 + 标注未完成 |
| **收敛型** | loop 不再进展 | 动作指纹重复、置信度阈值、评分达标 | 干预/收尾/换策略 |
| **错误型** | 不可恢复失败 | 连续失败次数、guardrail 触发、致命错误码 | 升级人工 / 降级返回 |
| **外部型** | 外部要求停止 | 用户取消、deadline、HITL 拒绝、需求变更 | 优雅取消 + checkpoint |

下面逐维度展开。

---

## 四、完成型：完成信号的三种实现与陷阱

### 4.1 三种做法

| 做法 | 机制 | 优点 | 陷阱 |
|------|------|------|------|
| **显式 finish 工具** | 给模型一个 `finish(answer)` 工具，调用即终止 | 语义最明确、可带结构化结果 | 模型可能忘记调（要配预算兜底） |
| **无 tool_call 即完成** | 模型这轮没调任何工具就视为给出最终答案 | 实现最简单，多数框架默认 | 有歧义：模型可能只是在"自言自语"思考 |
| **stop sequence / 特定 token** | 命中 `Final Answer:` 等标记就停 | 文本 ReAct 时代通用 | 最脆弱，模型格式漂移就失效 |

**推荐**：function-calling 模型用「无 tool_call 即完成」做默认，对需要结构化交付的场景再叠加一个显式 `finish` 工具，两者取先到。

### 4.2 finish 工具的 schema（带"完成度自评"）

让模型在收尾时自报完成度，便于上游判断是否假完成：

```json
{
  "name": "finish",
  "description": "任务已完成时调用，提交最终答案。未完成不要调用。",
  "parameters": {
    "type": "object",
    "properties": {
      "answer": {"type": "string"},
      "completed": {"type": "boolean", "description": "是否完全满足了用户目标"},
      "unresolved": {"type": "array", "items": {"type": "string"}, "description": "仍未解决的点"}
    },
    "required": ["answer", "completed"]
  }
}
```

### 4.3 完成信号的两个典型陷阱

- **假完成**：模型说"我已经处理好了"但其实没调工具、没产出。对策：完成判据外显（要求 `finish` 必须带可验证的 `answer`/证据），能用程序校验的就别信模型自述（见母文档第七节、《Agent工程细节与可靠执行》）。
- **该停不停**：模型一直"思考"不调 finish 也不给答案。对策：`tool_choice` 引导 + 预算兜底 + "无 tool_call 即完成"兜底。

## 五、预算型：分层预算闸

预算是**唯一能保证 loop 一定停**的硬闸，必须设。常见六个维度：

| 预算 | 含义 | 触顶处置 |
|------|------|----------|
| `max_steps` / `max_turns` | 最大迭代轮数 | 最常用，强制 finalize |
| `max_tokens` | 单次响应或整轮 token 上限 | 截断 + 收尾 |
| `max_cost` | 累计花费（美元） | 适合对外计费场景 |
| `max_wall_time` | 墙钟时间（deadline） | 适合有 SLA 的在线请求 |
| `max_rpm` | 每分钟请求数 | 限流，防止打爆下游 |
| `max_tool_calls` | 累计工具调用次数 | 防止工具滥用 |

**设计要点**：

1. **预算要分层**：单步预算（一次工具超时）+ 整轮预算（max_steps）+ 全局预算（cost/deadline）。
2. **deadline 要传递**：把 `ctx`（带 timeout）一路透传到工具调用，避免 loop 整体超时但单个工具还在死等。
3. **触顶必须标注**：返回结果要带 `reason="budget"` 和 `completed=false`，不能把半成品伪装成成品。

```python
class Budget:
    def __init__(self, max_steps=20, max_cost_usd=3, deadline_ts=None):
        self.max_steps, self.max_cost, self.deadline = max_steps, max_cost_usd, deadline_ts
    def exceeded(self, state) -> str | None:
        if state.step >= self.max_steps:    return "max_steps"
        if state.cost_usd >= self.max_cost: return "max_cost"
        if self.deadline and time.time() >= self.deadline: return "deadline"
        return None
```

## 六、收敛型：检测 loop 不再进展

预算闸只能"超时硬停"，但很多时候 loop 早就卡死、白白烧到触顶才停。收敛型停止能更早发现。

### 6.1 死循环检测（动作指纹）

对 `(tool_name, normalized_args)` 做指纹，维护近 N 轮指纹窗口，检测三种病态：

```python
def fingerprint(tool_call) -> str:
    return f"{tool_call.name}|{normalize(tool_call.args)}"  # 参数归一化后再指纹

def is_looping(fps: list[str]) -> bool:
    n = len(fps)
    # 1. 连续完全重复：同工具同参数刷 3 次
    if n >= 3 and fps[-1] == fps[-2] == fps[-3]:
        return True
    # 2. 短周期 A-B-A-B
    if n >= 4 and fps[-1] == fps[-3] and fps[-2] == fps[-4]:
        return True
    # 3. 高重复率：窗口内去重后占比过低
    if n >= 6 and len(set(fps[-6:])) <= 2:
        return True
    return False
```

Go（eino）版：

```go
func fingerprint(c schema.ToolCall) string {
	return c.Function.Name + "|" + normalize(c.Function.Arguments)
}

func isLooping(fps []string) bool {
	n := len(fps)
	if n >= 3 && fps[n-1] == fps[n-2] && fps[n-2] == fps[n-3] {
		return true // 连续完全重复
	}
	if n >= 4 && fps[n-1] == fps[n-3] && fps[n-2] == fps[n-4] {
		return true // 短周期 A-B-A-B
	}
	if n >= 6 && distinct(fps[n-6:]) <= 2 {
		return true // 高重复率
	}
	return false
}
```

**命中后的干预**（按强度递增）：
1. 注入提示："你在重复同一动作，请换思路，或基于已有信息直接收尾。"
2. 临时收紧：`tool_choice` 禁用刚重复的工具，或强制 `finish`。
3. 强制进入 finalize，返回"尽力而为"结果。

### 6.2 置信度 / 评分达标

- **置信度阈值**：让模型每轮自评置信度，达标即停（适合检索问答）。
- **评分达标**：有客观评估器（测试、校验器、LLM-judge）时，分数达标即停——这就是 Reflexion 外循环的停止条件（见母文档 6.2）。

## 七、错误型与外部型停止

### 7.1 错误型

- **连续失败计数**：同一 step 连续失败 N 次（如 3 次）就放弃该 step，升级人工或走 fallback，而不是无限自纠烧钱。
- **致命错误**：401/403（鉴权）、配额耗尽、工具不存在等不可恢复错误，立即停、不重试。
- **Guardrail 触发**：输入/输出护栏命中（如 prompt injection、违规内容）直接中断（见《安全与对齐》）。

### 7.2 外部型

- **用户取消**：通过 `ctx` 取消信号 / 中断 channel 优雅停止，保存 checkpoint。
- **Deadline**：在线请求超 SLA 立即收尾。
- **HITL 拒绝**：高风险动作审批被拒，终止或回退（见母文档 8.3）。
- **需求变更**：用户中途改需求，停当前 loop、合并新需求后决定续跑还是 replan（见《Agent工程细节与可靠执行》场景 9）。

## 八、主流产品 / 框架的停止机制（核心）

各家把上面五个维度落到了不同的 API 上。理解这张对照，就能在任何框架里快速找到"怎么让它停"。

### 8.1 LangChain AgentExecutor

封装好的 while 循环，停止参数直接挂在 executor 上：

```python
from langchain.agents import AgentExecutor

executor = AgentExecutor(
    agent=agent, tools=tools,
    max_iterations=15,             # 预算型：最多 15 轮（默认）
    max_execution_time=60,         # 预算型：墙钟 60s
    early_stopping_method="force", # 触顶收尾方式：force=返回固定停止语
)
```

- 正常终止：agent 返回 `AgentFinish`（即模型给出最终答案、不再要工具）。
- 触顶不抛异常，而是返回 `"Agent stopped due to iteration limit or time limit."`。
- `early_stopping_method`：`"force"` 直接返回停止语；`"generate"` 再让模型基于已有信息生成一次最终答案（不同版本支持度不同）。

### 8.2 LangGraph

把 loop 显式建成图，停止 = 走到 `END`，兜底 = `recursion_limit`：

```python
from langgraph.graph import StateGraph, END

def should_continue(state) -> str:
    last = state["messages"][-1]
    return "tools" if last.tool_calls else END   # 完成型：无 tool_call → END

graph.add_conditional_edges("agent", should_continue, {"tools": "tools", END: END})

app = graph.compile()
app.invoke(input, config={"recursion_limit": 50})  # 预算型：超 50 超步抛 GraphRecursionError
```

- 正常终止：条件边返回 `END`。
- 兜底：`recursion_limit`（默认 25，按"super-step"计数），触顶抛 `GraphRecursionError`。
- 外部型：`interrupt()` / `interrupt_before` 实现 HITL 暂停—恢复。

### 8.3 OpenAI Agents SDK

`Runner` 驱动的 loop，预算挂在 `run` 上：

```python
from agents import Agent, Runner
from agents.exceptions import MaxTurnsExceeded

try:
    result = await Runner.run(agent, input, max_turns=10)  # 预算型：默认 max_turns
except MaxTurnsExceeded:
    ...  # 触顶抛异常（注意：与 LangChain 返回停止语不同）
```

- 正常终止：agent 产出 `final_output`（设了 `output_type` 时即结构化输出；否则是没有工具/handoff 调用的纯文本）。
- 错误型：`InputGuardrailTripwireTriggered` / `OutputGuardrailTripwireTriggered`（护栏命中）。

### 8.4 Anthropic Claude（API + Agent SDK）

裸 API 的停止判据是 `stop_reason`：

```python
# 循环直到模型不再要工具
while resp.stop_reason == "tool_use":            # 完成型反信号
    tool_results = run_tools(resp.content)
    messages.append({"role": "user", "content": tool_results})
    resp = client.messages.create(
        model="claude-...", messages=messages, tools=tools,
        max_tokens=1024,                          # 预算型：单响应 token 上限
        stop_sequences=["<DONE>"],                # 完成型：命中即 stop_reason="stop_sequence"
    )
```

- `stop_reason` 取值：`end_turn`（正常说完）、`tool_use`（要调工具→继续）、`max_tokens`（被 token 截断）、`stop_sequence`（命中停止串）、`pause_turn`（长任务暂停，可续）。
- **Claude Agent SDK / Claude Code**：用 `max_turns` 选项给 agentic 循环设硬上限——这就是产品层把"预算型"暴露给用户的方式。

Go（eino）里没有 `stop_reason` 字段，等价判据是 `len(resp.ToolCalls) == 0`；用内置 ReAct agent 时，预算闸就是 `MaxStep`：

```go
agent, _ := react.NewAgent(ctx, &react.AgentConfig{
	ToolCallingModel: cm,
	ToolsConfig:      compose.ToolsNodeConfig{Tools: tools},
	MaxStep:          20, // 预算型：硬上限（默认 12），触顶返回最后一条消息
})
out, _ := agent.Generate(ctx, []*schema.Message{schema.UserMessage(goal)})
// 完成型：当某轮模型返回的 *schema.Message 没有 ToolCalls，loop 自然结束
```

### 8.5 AutoGen（AgentChat）

把停止抽象成可组合的 `TerminationCondition`，用 `|`(或) `&`(与) 拼：

```python
from autogen_agentchat.conditions import (
    MaxMessageTermination, TextMentionTermination,
    TokenUsageTermination, TimeoutTermination,
)

termination = (
    MaxMessageTermination(20)              # 预算型：最多 20 条消息
    | TextMentionTermination("TERMINATE")  # 完成型：出现 TERMINATE 文本
    | TokenUsageTermination(max_total_token=50000)  # 预算型：token 上限
    | TimeoutTermination(120)              # 预算型：墙钟 120s
)
team = RoundRobinGroupChat([...], termination_condition=termination)
```

- 设计亮点：**停止条件是一等对象，可声明式组合**——这是最值得学习的抽象。
- 还有 `ExternalTermination`（外部手动停）、`HandoffTermination`、`SourceMatchTermination`。

### 8.6 CrewAI

停止挂在 Agent / Crew 上：

```python
agent = Agent(
    role="researcher",
    max_iter=25,            # 预算型：单 agent 最大迭代（默认 25），触顶强制给最佳答案
    max_execution_time=120, # 预算型：墙钟
    max_rpm=10,             # 预算型：每分钟请求数（限流）
)
```

### 8.7 AutoGPT / 自主 agent

- `--continuous-limit N`：连续模式下的循环次数上限；非连续模式每轮要用户 y/n 确认（外部型 + HITL）。
- 预算型：API 花费上限。
- 这类"完全自主"agent 是停止条件最难设计的——靠多道预算 + 频繁人审兜底。

### 8.8 对照速查

| 框架/产品 | 正常终止 | 主要预算闸 | 触顶行为 |
|-----------|----------|------------|----------|
| LangChain AgentExecutor | `AgentFinish`（无工具） | `max_iterations` / `max_execution_time` | 返回停止语（不抛） |
| LangGraph | 条件边→`END` | `recursion_limit`(25) | 抛 `GraphRecursionError` |
| OpenAI Agents SDK | `final_output` | `max_turns` | 抛 `MaxTurnsExceeded` |
| Claude API | `stop_reason=end_turn` | `max_tokens` / 自管轮数 | 由你处理 |
| Claude Agent SDK | 同上 | `max_turns` | 停止并返回 |
| AutoGen | `TextMention`/条件 | `MaxMessage`/`TokenUsage`/`Timeout` | 条件命中即停 |
| CrewAI | 任务产出 | `max_iter` / `max_execution_time` | 强制给最佳答案 |
| eino ReAct | 无 `ToolCalls` | `MaxStep`(12) | 返回最后消息 |

**观察**：触顶行为有两派——**抛异常**（LangGraph、OpenAI SDK，逼你显式处理）vs **静默返回**（LangChain、CrewAI、eino，更顺滑但容易把半成品当成品）。生产里建议无论哪派，都在外层统一把"触顶"转成带 `completed=false` 的结构化结果。

## 九、组合多道闸：优先级与处置

五个维度要叠加成一个判定链，**顺序很重要**（先查最该早停的）：

```
每轮开始：
  1. 外部中断/取消？      → 立即优雅停，存 checkpoint     （最高优先）
  2. 预算耗尽？           → finalize(reason=budget)
  3. 上一轮致命错误?      → 升级人工 / 降级返回
  推理后：
  4. 显式 finish?         → finalize(success)
  5. 死循环 / 不收敛?     → 干预；连续命中则强制 finalize
  6. 连续失败超限?        → 放弃该 step / 升级
  否则：执行工具，进入下一轮
```

**每类停止的处置不同**，不能一律 `return`：

| 停止类型 | 处置 | 返回标注 |
|----------|------|----------|
| 完成型 | 返回结果 | `completed=true` |
| 预算型 | 强制收尾 | `completed=false, reason=budget` |
| 收敛型 | 先干预，无效再收尾 | `completed=false, reason=stuck` |
| 错误型 | 升级人工 / fallback | `completed=false, reason=error` |
| 外部型 | 保存 checkpoint 后停 | `completed=false, reason=cancelled` |

## 十、停止后的收尾（finalize）策略

触顶/中断后**不能直接抛错或返回空**，要给一个尽力而为的交付：

- **force**：返回固定停止语 + 已有的中间结果（最省，LangChain 默认）。
- **generate**：再让模型基于已积累的上下文生成一次"基于目前信息的最佳答案"（多花一次调用，体验更好）。
- **结构化收尾**：返回 `{answer, completed:false, done_steps, unresolved, reason}`，让上游能判断要不要续跑、要不要人工接管。

**核心结论**：停止不是终点，收尾质量决定了"被迫停止"是优雅降级还是直接翻车。

## 十一、eino（Go）中实现一套完整停止条件

把五个维度（完成 / 预算 / 收敛 / 错误 / 外部）装进一个生产可用的手写 loop。分四块讲：① 类型与预算（含成本核算）；② 判停辅助函数；③ 主循环；④ 并行执行与收尾。下列为示意性 eino 代码，需 `context`、`sync`、`time`、`encoding/json` 等包。

### 11.1 类型、预算与成本核算

```go
type StopReason string

const (
	StopSuccess   StopReason = "success"   // 完成型：finish / 无 tool_call
	StopBudget    StopReason = "budget"    // 预算型：steps / cost / deadline
	StopStuck     StopReason = "stuck"     // 收敛型：死循环
	StopError     StopReason = "error"     // 错误型：连续失败
	StopCancelled StopReason = "cancelled" // 外部型：取消 / 超时
)

type Result struct {
	Answer    string
	Completed bool       // 是否真正完成（绝不把被迫停止伪装成成功）
	Reason    StopReason
	Steps     int
	CostUSD   float64
}

// Budget 把预算型停止的多个维度收在一起
type Budget struct {
	MaxSteps   int
	MaxCostUSD float64
	Deadline   time.Time // 墙钟上限；零值表示不限
	InputPerM  float64   // 计价：美元 / 百万 input token
	OutputPerM float64   // 计价：美元 / 百万 output token
}

func (b Budget) Exceeded(step int, cost float64) bool {
	switch {
	case b.MaxSteps > 0 && step >= b.MaxSteps:
		return true
	case b.MaxCostUSD > 0 && cost >= b.MaxCostUSD:
		return true
	case !b.Deadline.IsZero() && time.Now().After(b.Deadline):
		return true
	default:
		return false
	}
}

// 从模型返回的 token usage 累加成本（作为预算闸的一维）
func (b Budget) addCost(prev float64, m *schema.Message) float64 {
	if m.ResponseMeta == nil || m.ResponseMeta.Usage == nil {
		return prev
	}
	u := m.ResponseMeta.Usage
	return prev +
		float64(u.PromptTokens)/1e6*b.InputPerM +
		float64(u.CompletionTokens)/1e6*b.OutputPerM
}
```

### 11.2 判停辅助函数

```go
// 动作指纹：工具名 + 归一化参数
func fingerprint(c schema.ToolCall) string {
	return c.Function.Name + "|" + normalizeArgs(c.Function.Arguments)
}

// 参数归一化：解析成 map 再序列化，消除键序/空白等无意义差异
func normalizeArgs(raw string) string {
	var v map[string]any
	if json.Unmarshal([]byte(raw), &v) != nil {
		return raw
	}
	b, _ := json.Marshal(v) // Go 的 map JSON 序列化按 key 排序，天然归一
	return string(b)
}

// 死循环检测：连续重复 / 短周期 A-B-A-B / 高重复率
func isLooping(fps []string) bool {
	n := len(fps)
	if n >= 3 && fps[n-1] == fps[n-2] && fps[n-2] == fps[n-3] {
		return true
	}
	if n >= 4 && fps[n-1] == fps[n-3] && fps[n-2] == fps[n-4] {
		return true
	}
	if n >= 6 {
		seen := map[string]struct{}{}
		for _, f := range fps[n-6:] {
			seen[f] = struct{}{}
		}
		if len(seen) <= 2 {
			return true
		}
	}
	return false
}

func findCall(m *schema.Message, name string) (schema.ToolCall, bool) {
	for _, c := range m.ToolCalls {
		if c.Function.Name == name {
			return c, true
		}
	}
	return schema.ToolCall{}, false
}

// 从 tool_call 参数里取某个字段（如 finish 的 answer）
func argString(c schema.ToolCall, key string) string {
	var m map[string]any
	_ = json.Unmarshal([]byte(c.Function.Arguments), &m)
	if s, ok := m[key].(string); ok {
		return s
	}
	return ""
}
```

### 11.3 主循环

```go
func runWithStops(ctx context.Context, chat model.ToolCallingChatModel,
	tools map[string]tool.InvokableTool, b Budget,
	history []*schema.Message) Result {

	fps := make([]string, 0, 16)
	var cost float64
	consecutiveFail, stuckNudges := 0, 0

	for step := 0; ; step++ {
		// ── 外部型：取消 / deadline ──
		if ctx.Err() != nil {
			return Result{Reason: StopCancelled, Steps: step, CostUSD: cost}
		}
		// ── 预算型 ──
		if b.Exceeded(step, cost) {
			return finalize(ctx, chat, history, StopBudget, step, cost)
		}

		// ── 推理（含错误型重试）──
		resp, err := chat.Generate(ctx, history)
		if err != nil {
			if consecutiveFail++; consecutiveFail >= 3 {
				return Result{Reason: StopError, Steps: step, CostUSD: cost}
			}
			time.Sleep(time.Duration(consecutiveFail) * time.Second) // 退避
			continue
		}
		consecutiveFail = 0
		cost = b.addCost(cost, resp)
		history = append(history, resp)
		// trace(step, resp, cost)  // 逐轮可观测，略

		// ── 完成型：显式 finish 或无 tool_call ──
		if c, ok := findCall(resp, "finish"); ok {
			return Result{Answer: argString(c, "answer"), Completed: true,
				Reason: StopSuccess, Steps: step, CostUSD: cost}
		}
		if len(resp.ToolCalls) == 0 {
			return Result{Answer: resp.Content, Completed: true,
				Reason: StopSuccess, Steps: step, CostUSD: cost}
		}

		// ── 收敛型：死循环检测，先劝导一次，再强制收尾（两段式）──
		fps = append(fps, fingerprint(resp.ToolCalls[0]))
		if isLooping(fps) {
			if stuckNudges++; stuckNudges >= 2 {
				return finalize(ctx, chat, history, StopStuck, step, cost)
			}
			history = append(history, schema.UserMessage(
				"你在重复同一动作，请换思路，或基于已有信息直接给出答案。"))
			continue
		}

		// ── 工具执行（并行）+ 回填 ──
		history = append(history, runToolsParallel(ctx, tools, resp.ToolCalls)...)
	}
}
```

### 11.4 并行执行与收尾（finalize）

```go
// 并行执行同一轮的多个 tool_call，按原顺序回填
func runToolsParallel(ctx context.Context, tools map[string]tool.InvokableTool,
	calls []schema.ToolCall) []*schema.Message {

	outs := make([]*schema.Message, len(calls))
	var wg sync.WaitGroup
	for i, c := range calls {
		wg.Add(1)
		go func(i int, c schema.ToolCall) {
			defer wg.Done()
			t, ok := tools[c.Function.Name]
			if !ok {
				outs[i] = schema.ToolMessage("ERROR: unknown tool "+c.Function.Name, c.ID)
				return
			}
			out, err := t.InvokableRun(ctx, c.Function.Arguments)
			if err != nil {
				out = "ERROR: " + err.Error()
			}
			outs[i] = schema.ToolMessage(out, c.ID)
		}(i, c)
	}
	wg.Wait()
	return outs
}

// 收尾：generate 策略——让模型基于已有上下文给"尽力而为"的答案，并显式标注未完成
func finalize(ctx context.Context, chat model.ToolCallingChatModel,
	history []*schema.Message, reason StopReason, step int, cost float64) Result {

	history = append(history, schema.UserMessage(
		"已达停止条件，请基于目前已掌握的信息直接给出当前最好的答案，不要再调用工具。"))
	answer := ""
	if resp, err := chat.Generate(ctx, history); err == nil { // 生产中此处应禁用工具
		answer = resp.Content
	}
	return Result{Answer: answer, Completed: false, Reason: reason, Steps: step, CostUSD: cost}
}
```

**要点**：
- 五类停止各有独立判定与处置，返回值用 `StopReason + Completed` 显式区分，绝不把"被迫停止"伪装成成功。
- 预算型 / 收敛型走 `finalize`（generate 策略），尽力而为而非直接返回空。
- 死循环检测**两段式**：先劝导一次、再强制收尾，避免一抖动就放弃。
- 工具**并行执行但顺序回填**；`ctx` 同时承载取消与 deadline。
- 成本由模型返回的 token usage 累加，作为预算闸的一维（`Budget` 三维：steps / cost / deadline）。

## 十三、思考题（含解答）

### Q1：只设 `max_steps` 一道闸够不够？
不够。`max_steps` 只能保证"最终会停"，但 loop 可能早就卡死，白白烧到触顶才停；也无法处理墙钟超时、成本超标、外部取消。至少要叠加：完成信号（正常终止）、死循环检测（更早发现卡死）、deadline/cost（在线/计费场景）、外部取消（用户体验）。

### Q2：为什么"触顶静默返回"比"抛异常"更危险？
静默返回（LangChain/CrewAI/eino）会把强制中断的半成品当成正常结果交回，上游若不检查 `reason/completed` 就会把它当成品用，造成"假完成"。抛异常（LangGraph/OpenAI SDK）则逼调用方显式处理。无论哪派，都应在外层把触顶统一转成带 `completed=false, reason=...` 的结构化结果。

### Q3：怎么区分"模型在认真思考"和"模型卡死了"？
靠动作指纹而非轮数。认真思考时每轮的 `(tool, args)` 在变化、信息在累积；卡死时表现为连续相同指纹、A-B-A-B 短周期、或窗口内去重率极低。检测到病态再注入提示/收紧 tool_choice/强制 finalize，而不是简单等 max_steps。

### Q4：AutoGen 的 `TerminationCondition` 抽象好在哪？
它把停止条件做成可声明式组合的一等对象，用 `|`/`&` 拼装（MaxMessage | TextMention | Timeout | TokenUsage）。好处：停止逻辑与执行逻辑解耦、可复用、可测试、可按场景灵活叠加，而不是把一堆 if 散落在 loop 里。这正是"组合多道闸"思想的最佳工程化范例。

### Q5：Claude 的 `stop_reason` 在 loop 里怎么用？
它是完成型信号的来源：`tool_use` 表示还要调工具（继续循环），`end_turn` 表示正常说完（终止），`max_tokens` 表示被预算截断（需判断是否要续写），`stop_sequence` 表示命中了你设的停止串。eino(Go) 没有这个字段，等价判据是 `len(resp.ToolCalls)==0`，产品层（Claude Agent SDK）则用 `max_turns` 暴露预算闸。

## 十四、核心要点速记

1. 停止难在两难：停不下来会失控，停太早会假完成。
2. 健壮 loop = 1 个精确完成信号 + N 道兜底闸（预算/收敛/错误/外部）。
3. 预算是唯一保证一定会停的硬闸，且必须分层（单步/整轮/全局）+ deadline 透传。
4. 死循环检测靠动作指纹（连续重复 / 短周期 / 高重复率），比等 max_steps 更早。
5. 主流框架停止 API 各异，但都能映射到五个维度；触顶分"抛异常 vs 静默返回"两派。
6. AutoGen 的可组合 `TerminationCondition` 是停止条件工程化的范本。
7. 停止后必须优雅收尾（force/generate）+ 结构化标注 `completed/reason`。
