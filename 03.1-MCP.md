# MCP（Model Context Protocol）协议详解

> 母文档：《[03-工具调用](./03-工具调用.md)》。
> 工具调用解决"模型怎么调一个工具"，MCP 解决"工具怎么被标准化地提供、跨应用复用"。这里把 MCP 讲透：动机、架构、能力原语、传输、与直接 function calling 的对比、生态、安全与局限。

---

## 一、为什么需要 MCP

### 1.1 它解决的问题：工具集成的碎片化

在 MCP 之前，每个 LLM 应用都要**自己实现一遍**工具集成：GitHub、Slack、数据库、文件系统……

- 重复造轮子：每个应用都写一遍 GitHub 工具。
- 无法复用：A 应用写的工具，B 应用用不了。
- 生态碎片化：M 个应用 × N 个工具 = M×N 套集成。

MCP 的目标是把它变成 **M + N**：工具方写一次 MCP server，所有支持 MCP 的应用都能用。

### 1.2 类比 USB-C
MCP 之于"LLM 应用 ↔ 外部工具/数据"，就像 USB-C 之于"设备 ↔ 外设"——**一个标准接口**，两边各自演进、即插即用。

**核心判断**：MCP 不是一种新的"工具调用方式"（模型侧还是 function calling），而是**工具集成的标准化传输协议**——把"工具怎么被发现、描述、调用、授权"标准化，解决的是工程集成问题，不是模型能力问题。

## 二、架构

```
┌─────────────────────────────┐         ┌──────────────────┐
│  Host（LLM 应用）            │         │  MCP Server      │
│  Claude Desktop / Cursor /  │         │  github / postgres│
│  自研 Agent                 │         │  / filesystem ... │
│   ┌──────────────┐          │ JSON-RPC│   ┌────────────┐ │
│   │  MCP Client  │ ◄────────┼─────────┼─► │  真实工具/  │ │
│   │ （每 server  │ stdio /  │ 2.0     │   │  数据源     │ │
│   │   一个）     │ HTTP     │         │   └────────────┘ │
│   └──────────────┘          │         │                  │
└─────────────────────────────┘         └──────────────────┘
```

| 角色 | 职责 |
|------|------|
| **Host** | 使用 LLM 的应用（Claude Desktop、Cursor、Cline、自研 Agent），管理一个或多个 client |
| **Client** | 嵌在 Host 内，与某个 Server 维持 1:1 连接，负责协议通信 |
| **Server** | 暴露 tools / resources / prompts，连接真实工具与数据，可本地或远程 |

通信用 **JSON-RPC 2.0**，初始化时双方 **capability negotiation**（协商各自支持哪些能力）。

## 三、能力原语

### 3.1 Server 暴露给应用的三大原语
| 原语 | 谁控制 | 说明 | 例子 |
|------|--------|------|------|
| **Tools** | 模型（model-controlled） | 可被 LLM 调用执行的函数 | `create_issue`、`run_sql` |
| **Resources** | 应用（app-controlled） | LLM 可读取的只读数据，有 URI | 文件、数据库表、API 响应 |
| **Prompts** | 用户（user-controlled） | 预设的提示模板/工作流 | "code review""bug triage" |

三者的区别在于"**谁决定用它**"：Tools 由模型自主调用，Resources 由应用决定塞什么进上下文，Prompts 由用户主动触发（如 slash command）。

### 3.2 反向能力（Server 反过来用 Client 的）
MCP 不只是 Server→Client 单向暴露，Client 也能向 Server 提供能力：
- **Sampling**：Server 可以反过来请求 Host 的 LLM 做一次补全（让 server 内部也能用模型，但由 host 控制模型与计费）。
- **Roots**：Host 告诉 Server 可访问的文件系统根，划定边界。
- **Elicitation**（较新）：Server 运行中可请求用户补充输入。

**面试常考点**：能说出"Tools/Resources/Prompts 分别由模型/应用/用户控制"，以及"Sampling 是 server 反向调 host 模型"，说明你真读过协议而不只听过名字。

## 四、传输（Transport）

| 传输 | 场景 | 说明 |
|------|------|------|
| **stdio** | 本地 server | Host 把 server 当子进程拉起，走标准输入输出，最简单、低延迟 |
| **Streamable HTTP** | 远程 server | 2025-03 规范引入，取代早期的 HTTP+SSE，单端点、支持流式与可恢复连接 |

- 早期（2024 末首版）远程用 **HTTP + SSE**，后演进为 **Streamable HTTP**，更适合无状态/可扩展部署。
- 远程 server 的鉴权：规范后续补充了 **OAuth 2.1** 授权框架。

---

## 五、MCP vs 直接 function calling

**关键澄清**：MCP 和 function calling 不是二选一——模型侧仍然是 function calling，MCP 只是改变了"工具从哪来、怎么被提供"。

| 维度 | 直接定义 tools | 通过 MCP |
|------|----------------|----------|
| 耦合 | 紧耦合，工具代码在应用内 | 松耦合，工具在独立 server 进程 |
| 复用 | 单应用内 | 跨所有 MCP host 复用 |
| 隔离 | 同进程 | 独立进程，权限可显式授权 |
| 复杂度 | 低，直接写 | 高一些，要起 server、走 IPC |
| 性能 | 无额外开销 | 有 IPC/网络开销 |

**何时用 MCP**：
- 公司有**多个 Agent 应用**要复用同一批工具 → 用 MCP。
- 想接入**现成生态**（GitHub、Postgres 等社区 server）→ 用 MCP。
- 想让工具方与应用方**解耦分工** → 用 MCP。

**何时直接 tools 就够**：
- 单一应用、工具是私有的、就这一处用 → 直接 function calling 更简单，别为了用 MCP 而用。

## 六、生态与 SDK

- **社区 server**：GitHub、GitLab、Slack、Postgres、SQLite、文件系统、Puppeteer/Playwright、Brave Search 等大量现成 server。
- **Host**：Claude Desktop（首发）、Cursor、Cline、Continue、Zed 等；2025 年 OpenAI、Google 也宣布支持 MCP，事实上的跨厂商标准。
- **SDK**：Python、TypeScript、Java、Kotlin、C# 等官方 SDK，写 server 很快。

## 七、安全（重点，常被忽略）

MCP 引入"第三方 server"后，带来新的攻击面：

| 风险 | 说明 | 对策 |
|------|------|------|
| **恶意 server** | 装了不可信 server，能读数据/执行动作 | 只装可信来源、最小授权、审查 server |
| **工具投毒（tool poisoning）** | server 在工具 description 里藏恶意指令，诱导模型做坏事（prompt injection 的一种） | 审查工具描述、隔离不可信 server、输出护栏 |
| **过度授权** | server 拿到过多权限（整个文件系统） | Roots 限定范围、最小权限、敏感动作 HITL |
| **令牌/凭据泄露** | 远程 server 的 OAuth token 管理不当 | 规范的 OAuth 2.1、令牌隔离 |
| **供应链** | server 依赖被污染 | 锁版本、可信 registry |

**核心结论**：MCP 把工具变成可插拔的第三方组件，便利的同时把"信任边界"问题放大了——装一个 MCP server ≈ 给 agent 接了一个外部能力，必须按对待外部代码的标准做授权、隔离和审查（呼应《安全与对齐》《18-模型固有局限与工程兜底》的信任边界）。

## 八、局限

- 协议仍在快速演进（2024 末首发，2025 多次更新传输与鉴权），生态工具质量参差。
- 仍需 Host 端做工具发现、调度、选择——**MCP 不解决"模型选错工具"这个模型层问题**。
- 引入 IPC/网络开销，极致低延迟场景要权衡。
- 多 server 时的工具治理（数量爆炸、命名冲突、权限）仍要应用方自己管（见《03-工具调用》第五节）。

## 九、思考题（含解答）

### Q1：MCP 解决了什么问题？和直接定义 tools 比价值在哪？
解决工具集成的碎片化：MCP 之前每个应用都自己实现 GitHub/Slack/DB 工具，无法复用（M×N 套集成）。MCP 用标准协议（JSON-RPC + Tools/Resources/Prompts 原语）让工具方写一次 server、所有 host 复用（M+N）。相比直接 tools：松耦合（工具在独立 server）、跨应用复用、进程隔离可授权。代价是多了 IPC 开销和起 server 的复杂度。

### Q2：MCP 和 function calling 是替代关系吗？
不是。模型侧仍然是 function calling——MCP 不改变模型怎么调工具，只标准化"工具从哪来、怎么被发现/描述/授权"。可以理解为：function calling 是模型能力，MCP 是工具供给的传输协议，二者配合使用。

### Q3：MCP 的三种原语分别由谁控制？
Tools 由**模型**控制（模型自主决定调用）、Resources 由**应用**控制（应用决定把哪些只读数据塞进上下文）、Prompts 由**用户**控制（用户主动触发的模板/工作流，如 slash command）。这个"谁控制"的区分是协议设计的核心。

### Q4：MCP 带来了哪些新的安全风险？
引入第三方 server 放大了信任边界问题：恶意 server、工具投毒（在工具描述里藏注入指令）、过度授权、远程 server 的令牌泄露、依赖供应链。对策：只装可信 server、最小授权 + Roots 限范围、审查工具描述、敏感动作 HITL、规范 OAuth、输出护栏。装一个 server 要按"接入外部代码"的标准对待。

### Q5：什么时候不该用 MCP？
单一应用、工具私有、只这一处用时，直接 function calling 更简单，别为用 MCP 而用——会平添起 server、走 IPC 的复杂度和延迟。MCP 的价值在多应用复用与接入现成生态，没有这两个诉求时收益不明显。

## 十二、核心要点速记

1. MCP 解决工具集成碎片化：写一次 server，所有 host 复用，M×N → M+N。
2. 它是工具供给的传输协议，不是新的工具调用方式——模型侧仍是 function calling。
3. 架构：Host / Client / Server，JSON-RPC 2.0，stdio（本地）/ Streamable HTTP（远程）。
4. 三原语：Tools（模型控制）/ Resources（应用控制）/ Prompts（用户控制）；Sampling 是反向调 host 模型。
5. 多应用复用、接现成生态才用 MCP；单应用私有工具直接 tools 更简单。
6. 引入第三方 server 放大信任边界：恶意 server、工具投毒、过度授权要重点防。
7. MCP 不解决"模型选错工具"，工具治理仍在应用方。
